# TWT-Link 私有化部署方案

## Context

TWT-Link 是一款私有化部署的即时通讯软件，面向企业客户提供独立部署的聊天解决方案。每个客户在自己的服务器上独立运行整套系统，我们不提供公有云 SaaS 服务。

TWT-Link 与 TWT Chat（web-agent、web-customer 等客服系统）是两个独立产品线。本文档仅涉及 TWT-Link 的私有化部署，不包含 Chat 相关功能。

### TWT-Link 产品组成

| 端 | 平台 | 说明 |
|------|------|------|
| 桌面客户端 | macOS、Windows | 原生桌面应用，日常聊天主入口 |
| 移动端 APP | iOS、Android | 移动端应用，随时随地沟通 |
| 后台管理端 | Web | 组织管理、用户管理、权限配置 |
| 部署管理后台 | Web | 客户部署状态、版本管理、License 管理 |

## 一、当前部署现状

### 前端应用清单（Link 相关）

| 应用 | 包名 | 说明 |
|------|------|------|
| twt-link | @twt/twt-link | Link 产品原型与 Web 端 |
| deploy-admin | @twt/deploy-admin | 部署管理后台 |

### 客户端应用

| 应用 | 平台 | 说明 |
|------|------|------|
| Link Desktop | macOS / Windows | 桌面客户端（Electron 或原生） |
| Link Mobile | iOS / Android | 移动端 APP |

### 现有部署流程（Web 端）
```bash
pnpm build:${app}    # Vite 构建，base path 为 /<app-name>/
pnpm deploy:${app}   # 构建 + rsync 到远程服务器 + 触发 webhook
```

部署脚本 `apps/upcode.sh` 通过 rsync 将 `dist/` 同步到远程服务器，部署后触发 webhook 进行自动部署。

### 现有配置机制
`@twt/branding` 的 `loadTenantConfig()` 在生产环境根据 hostname 从 `/tenants/{hostname}.json` 加载客户配置，开发环境返回默认配置。配置内容包括品牌信息、APP 配置、功能开关等。

## 二、私有化部署架构

### 目标
为客户提供一键部署的 Link 服务端，客户端（桌面 + 移动端）通过配置服务器地址连接。

### 客户侧部署（Docker 容器化）
客户在自己的服务器上通过 Docker Compose 拉起 Link 服务端：
```
docker-compose.yml（客户服务器）
├── nginx          # 反向代理
├── api-server     # Link 后端 API 服务（消息、用户、组织等）
├── im-server      # IM 长连接服务（WebSocket / TCP）
├── database       # PostgreSQL / MySQL
├── redis          # 缓存 + 会话 + 消息队列
├── file-server    # 文件存储服务（头像、聊天文件等）
└── worker         # 异步任务（推送通知、离线消息等）
```

### 我们侧管理工具
- deploy-admin（部署管理后台）是我们内部使用的 Web 管理工具，用于管理客户部署状态、License、版本等
- deploy-admin 沿用现有的 Vite 构建 + rsync 部署方式，部署在我们自己的服务器上，不交付给客户
- 客户端安装包（桌面 / 移动端）通过 deploy-admin 管理分发

### 交付模式：在线交付

采用在线交付模式，通过私有 Docker Registry 分发镜像。

**交付流程：**
1. 我们在 CI/CD 中构建 Docker 镜像，推送到私有 Registry（如阿里云 ACR）
2. 在 deploy-admin 中为客户创建部署记录，选择版本，生成客户专属的 `.env` 配置文件和 `docker-compose.yml`
3. 将部署文件（docker-compose.yml + .env + 初始化脚本）提供给客户
4. 客户在服务器上执行：
   ```bash
   # 登录私有 Registry（我们提供凭证）
   docker login registry.twt-link.com

   # 拉取镜像并启动
   docker compose up -d
   ```

**deploy-admin 在交付中的角色：**
- 管理每个客户对应的版本号和 Registry 访问凭证
- 一键生成客户专属的部署配置文件（.env 预填客户信息）
- 追踪客户当前部署版本，推送升级通知

### 部署策略
- 客户服务端通过 Docker 镜像交付，客户执行 `docker compose up -d` 即可启动
- 客户端（桌面 + 移动端）通过安装包分发，不在 Docker 中
- 客户配置通过环境变量注入到 Docker 容器中
- Nginx 配置模板化，启动时根据环境变量生成

### 需要新建的文件
```
deploy/
├── docker-compose.yml          # 客户侧编排文件模板
├── docker-compose.prod.yml     # 生产覆盖
├── nginx/
│   ├── nginx.conf.template     # Nginx 配置模板
│   └── entrypoint.sh           # 启动脚本（环境变量替换）
├── .env.example                # 环境变量模板
└── scripts/
    ├── init.sh                 # 初始化脚本
    ├── upgrade.sh              # 升级脚本
    └── backup.sh               # 备份脚本
```

## 三、客户端分发与连接

### 桌面客户端（macOS / Windows）
- 通过安装包分发（.dmg / .exe），不走 Docker
- 首次启动时配置服务器地址（指向客户私有化部署的 API）
- 支持自动更新：客户端定期检查版本服务器，下载增量更新包

### 移动端 APP（iOS / Android）
- iOS 通过企业签名或 TestFlight 分发，Android 通过 APK 直接安装
- 同样通过配置服务器地址连接到私有化后端
- 推送通知：iOS 走 APNs，Android 走 FCM 或厂商推送通道

### 客户端配置
客户端首次连接时需要配置：
```json
{
  "serverUrl": "https://link.customer.com",
  "wsUrl": "wss://link.customer.com/ws"
}
```

可通过扫码、链接或手动输入方式完成配置。

- 后端配置：存储密钥、推送证书、数据库连接 → 环境变量


## 五、客户管理体系

### 客户生命周期
1. **签约** → 在 deploy-admin 创建客户记录，生成 License Key
2. **配置** → 收集客户的存储/推送/域名信息，生成客户专属配置
3. **部署** → 客户服务器上执行 `docker compose up -d`
4. **分发客户端** → 提供桌面安装包和移动端 APP，配置服务器地址
5. **验证** → 检查所有服务连通性，客户端能正常登录和收发消息
6. **交付** → 提供管理员账号和运维文档
7. **运维** → 版本升级（服务端 + 客户端）、监控、续费

### 客户信息管理（deploy-admin）
在现有 deploy-admin 基础上管理：
- 部署状态（待部署/运行中/已暂停/已下线）
- 服务端当前版本号
- 客户端最新版本号（桌面 / 移动端）
- License 到期时间
- 外部服务配置状态（存储/推送 是否已配置且连通）
- 服务器信息（IP、端口等）

## 六、License 授权管理

### 防破解方案

**签名机制：**
- License 使用 ECDSA（P-256）非对称签名，私钥仅存于我们的签发服务器，客户部署的后端镜像中只内嵌公钥
- 客户无法伪造或篡改 License 内容（修改任何字段都会导致签名校验失败）
- 签名算法和验证逻辑编译在 Go 二进制中，不以源码形式交付，逆向成本高

**绑定策略：域名或 IP 绑定**

License 支持两种绑定模式，签发时根据客户实际情况选择：

| 绑定模式 | 适用场景 | License 字段 | 换服务器影响 |
|------|------|------|------|
| 域名绑定 | 客户有域名（公网或内网 DNS） | `"bindType": "domain", "bindValue": "link.customer.com"` | 域名不变则无需重新签发 |
| IP 绑定 | 客户无域名，直接用 IP 访问（常见于内网部署） | `"bindType": "ip", "bindValue": "192.168.1.100"` | 换 IP 需重新签发 |

- 推荐优先使用域名绑定，迁移成本最低
- 内网客户如果没有内部 DNS，使用 IP 绑定；内网服务器通常稳定，换 IP 频率低
- 后端启动时根据 `bindType` 决定校验方式：域名模式校验当前请求域名，IP 模式校验本机网卡 IP
- IP 绑定支持填写多个 IP（如 `"bindValue": "192.168.1.100,192.168.1.101"`），适配双网卡或主备切换场景
- 不绑定硬件指纹（MAC/CPU ID），避免云服务器重建、网卡变更等场景导致意外失效

**在线心跳验证：**
- 后端每 24 小时向我们的授权服务器发送心跳，上报当前域名和服务器 IP
- 授权服务器检测到同一 License 从多个不同 IP 同时心跳时，触发告警（防止 License 被复制到多台服务器）
- 断网宽限期 7 天：超过 7 天未成功心跳，服务自动降级为只读模式（可查看历史消息，不能发送新消息）
- 宽限期内恢复网络后自动恢复正常

**离线验证降级：**
- 即使无法联网，后端仍可通过公钥验证 License 签名的基本有效性（到期时间、绑定值匹配）
- 离线模式下无法检测 License 复用，因此功能降级：禁用视频通话、屏幕共享等高级功能

### 客户换服务器处理

| 场景 | 绑定模式 | 是否需要重新签发 | 操作 |
|------|------|------|------|
| 换服务器，域名不变 | 域名 | 不需要 | 新服务器 `docker compose up -d`，自动生效 |
| 换服务器，域名也换 | 域名 | 需要 | deploy-admin 更新域名，重新签发 |
| 换服务器，IP 变了 | IP | 需要 | deploy-admin 更新 IP，重新签发 |
| 换服务器，IP 在多 IP 列表内 | IP（多值） | 不需要 | 新 IP 已在绑定列表中，自动生效 |
| 同域名多服务器（负载均衡） | 域名 | 不需要 | 心跳识别为同域名多实例，不触发告警 |
| License 复制到未绑定的域名/IP | 任意 | 自动失效 | 绑定值不匹配，后端拒绝启动 |

### 方案
- 每个客户分配唯一 License Key
- License 包含：客户 ID、绑定类型（域名/IP）、绑定值、到期时间、功能权限、用户数上限、ECDSA 签名
- 后端启动时校验 License（签名 + 绑定值匹配 + 到期时间），定期向授权服务器心跳验证
- 离线场景：License 内嵌签名，可离线验证基本有效性，但功能降级

### License 结构
```json
{
  "customerId": "customer-001",
  "bindType": "domain",
  "bindValue": "link.customer.com",
  "plan": "professional",
  "maxUsers": 500,
  "features": ["group-chat", "file-sharing", "video-call", "screen-sharing"],
  "issuedAt": "2026-04-24",
  "expiresAt": "2027-04-22",
  "signature": "MEUCIQD...base64..."
}
```

IP 绑定示例：
```json
{
  "customerId": "customer-002",
  "bindType": "ip",
  "bindValue": "192.168.1.100,192.168.1.101",
  "plan": "professional",
  "maxUsers": 300,
  "features": ["group-chat", "file-sharing", "video-call"],
  "issuedAt": "2026-04-24",
  "expiresAt": "2027-04-22",
  "signature": "MEUCIQD...base64..."
}
```

### 验证流程
```
后端启动
  ├── 1. 读取 License 文件
  ├── 2. 用内嵌公钥验证 ECDSA 签名 → 失败则拒绝启动
  ├── 3. 根据 bindType 校验绑定值
  │     ├── domain → 校验当前请求域名是否匹配 bindValue
  │     └── ip → 校验本机网卡 IP 是否在 bindValue 列表中
  │     → 不匹配则拒绝启动
  ├── 4. 校验到期时间 → 已过期则拒绝启动
  └── 5. 启动成功，开始 24h 周期心跳
        ├── 心跳成功 → 正常运行
        ├── 心跳失败 → 进入宽限期倒计时（7 天）
        └── 宽限期到期 → 降级为只读模式
```

### License 状态与客户端行为

后端通过 API 向客户端下发当前 License 状态，客户端根据状态调整 UI 和功能可用性。

**License 状态定义：**

| 状态 | 触发条件 | 后端行为 |
|------|------|------|
| `normal` | License 有效且心跳正常 | 全部功能可用 |
| `grace_period` | 心跳失败，宽限期内（≤7 天） | 全部功能可用，后端在 API 响应头中附带 `X-License-Status: grace` 和 `X-License-Grace-Remaining: 5d` |
| `degraded` | 宽限期到期，或离线验证模式 | 核心功能可用，高级功能禁用，API 返回 `X-License-Status: degraded` |
| `expired` | License 到期时间已过 | 后端拒绝启动，客户端无法连接 |

**客户端（桌面 + APP）处理逻辑：**

客户端在每次 API 请求时读取响应头中的 License 状态，据此调整界面：

1. `normal` — 正常状态
   - 所有功能正常使用，无任何提示

2. `grace_period` — 宽限期
   - 桌面端：顶部出现黄色通知条，显示「服务授权验证异常，剩余 X 天恢复期，请联系管理员」，可关闭但每次启动重新出现
   - APP 端：首页顶部显示同样的黄色提示条，不可关闭
   - 所有功能仍然正常可用，不影响使用体验
   - 仅管理员角色额外显示「请检查服务器网络连接」的提示

3. `degraded` — 降级模式
   - 桌面端和 APP 端：顶部出现红色通知条，显示「服务已降级，部分功能不可用，请联系管理员恢复」，不可关闭
   - 功能限制：
     - 可用：查看历史消息、浏览联系人列表、查看文件列表
     - 禁用（按钮置灰 + 点击弹出提示）：发送新消息、发起视频/语音通话、屏幕共享、上传文件、创建群组
   - 禁用功能点击时弹出统一提示：「当前服务已降级，此功能暂不可用。请联系管理员恢复服务授权。」
   - APP 推送通知停止（后端不再触发推送）

4. 连接失败 — 后端未启动（License 过期或校验失败）
   - 客户端无法建立连接，显示全屏错误页：「无法连接到服务器，请检查网络或联系管理员」
   - 提供「重试」按钮和「切换服务器」入口
   - 桌面端：本地已缓存的历史消息仍可离线浏览（只读）
   - APP 端：同上，已缓存的消息可离线查看

**客户端状态检测机制：**
- 客户端每次 API 请求自动读取 `X-License-Status` 响应头，无需额外轮询
- WebSocket 长连接场景：后端在 License 状态变更时主动推送 `license_status_changed` 事件，客户端实时响应
- 客户端本地缓存最后一次 License 状态，断网时沿用缓存状态（避免断网就立刻降级）

## 七、软件升级策略

### 服务端升级
1. 我们发布新版 Docker 镜像到私有 Registry
2. 通知客户有新版本可用
3. 客户执行升级脚本：
   ```bash
   ./scripts/upgrade.sh v2.5.0
   # 拉取新镜像 → 备份数据 → 数据库迁移 → 滚动更新
   ```
4. 支持回滚：`./scripts/upgrade.sh --rollback`

### 客户端升级
- 桌面客户端：内置自动更新机制，检测到新版本后提示用户更新
- 移动端 APP：通过应用内更新提示，引导用户下载新版本
- 客户端版本与服务端版本需保持兼容，服务端 API 保持向后兼容

### 版本管理
- 语义化版本号（SemVer）
- 服务端和客户端独立版本号，通过兼容性矩阵管理
- 维护 CHANGELOG
- 重大版本提前通知，提供迁移指南
- 客户可以选择跳过小版本，但大版本必须按序升级

## 八、安全注意事项

### 凭证安全
- 将 `TenantConfig` 中的敏感凭证从前端配置中移除，改为后端环境变量
- 所有敏感信息通过环境变量注入，不进入代码仓库或前端可访问的文件
- Docker secrets 或 Vault 管理生产凭证

### 通信安全
- 客户端与服务端通信全程加密（TLS）
- IM 消息传输加密，可选端到端加密（E2EE）
- API 通信使用 JWT
- 文件传输加密

### 数据隔离
- 每个客户独立部署，物理隔离，天然不存在数据交叉
- 文件存储隔离：每个客户使用自己的 bucket

### 客户端安全
- 桌面客户端代码签名（macOS notarization / Windows code signing）
- 移动端 APP 签名验证
- 本地数据加密存储（聊天记录、登录凭证等）

### 代码安全
- 交付编译后的 Docker 镜像和客户端安装包，不交付源码
- 管理后台前端代码经过构建压缩

## 九、实施路径（分阶段）

### 阶段 1：配置安全化（1-2 周）
- 将存储/推送等敏感凭证从 `TenantConfig` 前端配置中剥离
- 后端增加配置 API，前端只获取非敏感的配置
- 修改 `packages/branding/src/types.ts`，移除敏感字段

### 阶段 2：服务端 Docker 化（2-3 周）
- 编写客户侧 docker-compose.yml（含 API 服务、IM 服务、Nginx、数据库、Redis）
- 编写 Nginx 配置模板和启动脚本
- 编写 `.env.example` 和初始化脚本
