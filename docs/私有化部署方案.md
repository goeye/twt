# TWT-Link 私有化部署方案

## Context

TWT-Link 是一套私有化部署的客服系统。每个客户在自己的服务器上独立运行整套系统，我们不提供公有云 SaaS 服务。

当前部署方式是通过 rsync 将构建产物同步到服务器，由 Nginx 提供静态文件服务。前端是 pnpm monorepo（10 个 Vue 3 应用），后端服务不在此仓库中。

现有的 `@twt/branding` 包提供了基于域名的客户配置加载机制，可作为私有化配置注入的基础。

## 一、当前部署现状

### 前端应用清单
| 应用 | 包名 | 说明 |
|------|------|------|
| web-agent | @twt/web-agent | 客服工作台 |
| web-customer | @twt/web-customer | 客户端 |
| web-admin | @twt/web-admin | 管理后台 |
| mobile-app | @twt/mobile-app | 移动应用 |
| web-website | @twt/web-website | 官网 |
| docs | @twt/docs | 文档站 |
| twt-admin | @twt/twt-admin | TWT 平台管理 |
| reseller-admin | @twt/reseller-admin | 经销商管理 |
| twt-link | @twt/twt-link | 链接分享工具 |
| deploy-admin | @twt/deploy-admin | 部署管理后台 |

### 现有部署流程
```bash
pnpm build:${app}    # Vite 构建，base path 为 /<app-name>/
pnpm deploy:${app}   # 构建 + rsync 到远程服务器 + 触发 webhook
```

部署脚本 `apps/upcode.sh` 通过 rsync 将 `dist/` 同步到 `121.40.183.14:8730`，部署后触发 `http://121.40.183.14:8090/hook` 进行自动部署。

### 现有配置机制
`@twt/branding` 的 `loadTenantConfig()` 在生产环境根据 hostname 从 `/tenants/{hostname}.json` 加载客户配置，开发环境返回默认配置。配置内容包括品牌信息、APP 配置、官网配置、邮件配置、OSS 配置和功能开关。

## 二、Docker 化部署架构

### 目标
将现有的 rsync 部署升级为 Docker 容器化部署，让客户可以一键拉起整套系统。

### 整体结构
```
docker-compose.yml
├── nginx          # 反向代理 + 静态文件服务（所有前端应用）
├── api-server     # 后端 API 服务
├── database       # PostgreSQL / MySQL
├── redis          # 缓存 + 会话
└── worker         # 异步任务（邮件发送、webhook 等）
```

### 前端打包策略
- 所有前端应用统一构建为一个 Nginx Docker 镜像
- 利用现有的 Vite base path 约定（`/<app-name>/`），所有应用共存于一个 Nginx 下
- 客户配置通过 `/tenants/{hostname}.json` 静态文件注入，与现有 `loadTenantConfig()` 机制兼容
- Nginx 配置模板化，启动时根据环境变量生成

### 需要新建的文件
```
deploy/
├── docker-compose.yml          # 编排文件
├── docker-compose.prod.yml     # 生产覆盖
├── nginx/
│   ├── Dockerfile              # 前端镜像
│   ├── nginx.conf.template     # Nginx 配置模板
│   └── entrypoint.sh           # 启动脚本（环境变量替换）
├── .env.example                # 环境变量模板
└── scripts/
    ├── init.sh                 # 初始化脚本
    ├── upgrade.sh              # 升级脚本
    └── backup.sh               # 备份脚本
```

## 三、配置注入方案

### 前端配置（客户可见）
沿用现有 `TenantConfig` 机制，扩展以下字段：

```typescript
// 在 packages/branding/src/types.ts 中扩展
interface TenantConfig {
  // ... 现有字段 ...

  // 新增：API 地址（指向客户自己的后端）
  apiBaseUrl: string
  wsBaseUrl: string        // WebSocket 地址
}
```

### 敏感配置剥离
当前 `TenantConfig` 中的 `oss.accessKeyId/Secret` 和 `email.smtpConfig` 暴露在前端 JSON 中，这是安全隐患。需要拆分：

- 前端配置：品牌、UI、域名、功能开关 → `/tenants/{hostname}.json`
- 后端配置：OSS 密钥、SMTP 密码、数据库连接 → 环境变量

```env
# .env.example（私有化部署环境变量）

# === 基础 ===
DOMAIN=chat.customer.com
API_PORT=8080

# === 数据库 ===
DB_HOST=localhost
DB_PORT=5432
DB_NAME=twt_chat
DB_USER=twt
DB_PASSWORD=

# === OSS（客户自备）===
OSS_PROVIDER=aliyun          # aliyun | aws | qiniu
OSS_BUCKET=
OSS_REGION=
OSS_ACCESS_KEY_ID=
OSS_ACCESS_KEY_SECRET=

# === 邮箱（客户自备）===
SMTP_HOST=
SMTP_PORT=465
SMTP_USER=
SMTP_PASSWORD=
EMAIL_FROM_NAME=
EMAIL_FROM_ADDRESS=

# === 许可证 ===
LICENSE_KEY=
```

## 四、客户管理体系

### 客户生命周期
1. **签约** → 在 twt-admin 创建客户记录，生成 License Key
2. **配置** → 收集客户的 OSS/邮箱/域名信息，生成客户专属配置
3. **部署** → 客户服务器上执行 `docker compose up -d`
4. **验证** → 检查所有外部服务连通性
5. **交付** → 提供管理员账号和运维文档
6. **运维** → 版本升级、监控、续费

### 客户信息管理（twt-admin 扩展）
在现有 twt-admin 基础上，增加：
- 部署状态（待部署/运行中/已暂停/已下线）
- 当前版本号
- License 到期时间
- 外部服务配置状态（OSS/邮箱 是否已配置且连通）
- 服务器信息（IP、SSH 端口等）

## 五、License 授权管理

### 方案
- 每个客户分配唯一 License Key
- License 包含：客户 ID、到期时间、功能权限、坐席上限
- 后端启动时校验 License，定期向授权服务器验证
- 离线场景：License 内嵌签名，可离线验证基本有效性，但功能降级

### License 结构
```json
{
  "customerId": "customer-001",
  "plan": "professional",
  "maxAgents": 50,
  "features": ["ai-agent", "multi-channel", "analytics"],
  "expiresAt": "2027-04-22",
  "signature": "..."
}
```

## 六、软件升级策略

### 利用现有版本检测机制
项目已有 `vite-plugin-version` 生成 `version.json`，各应用的 `useVersionCheck` 会轮询检测新版本。这套机制在私有化部署中同样适用。

### 升级流程
1. 我们发布新版 Docker 镜像到私有 Registry
2. 通知客户有新版本可用
3. 客户执行升级脚本：
   ```bash
   ./scripts/upgrade.sh v2.5.0
   # 拉取新镜像 → 备份数据 → 数据库迁移 → 滚动更新
   ```
4. 支持回滚：`./scripts/upgrade.sh --rollback`

### 版本管理
- 语义化版本号（SemVer）
- 维护 CHANGELOG
- 重大版本提前通知，提供迁移指南
- 客户可以选择跳过小版本，但大版本必须按序升级

## 七、安全注意事项

### 凭证安全
- 将 `TenantConfig` 中的 `oss.accessKeyId/Secret` 和 `email.smtpConfig` 从前端配置中移除，改为后端环境变量
- 所有敏感信息通过环境变量注入，不进入代码仓库或前端可访问的文件
- Docker secrets 或 Vault 管理生产凭证

### 数据隔离
- 每个客户独立部署，物理隔离，天然不存在数据交叉
- OSS 隔离：每个客户使用自己的 bucket

### 网络安全
- 强制 HTTPS
- API 通信使用 JWT
- 建议客户配置防火墙规则
- License 验证通信加密签名

### 代码安全
- 交付编译后的 Docker 镜像，不交付源码
- 前端代码经过 Vite 构建压缩，但本质上前端代码客户可见，不应包含核心业务逻辑

## 八、实施路径（分阶段）

### 阶段 1：配置安全化（1-2 周）
- 将 OSS/SMTP 敏感凭证从 `TenantConfig` 前端配置中剥离
- 后端增加配置 API，前端只获取非敏感的配置
- 修改 `packages/branding/src/types.ts`，移除敏感字段

### 阶段 2：Docker 化（2-3 周）
- 编写前端 Nginx Dockerfile（统一构建所有应用）
- 编写 docker-compose.yml（含后端、数据库、Redis）
- 编写 `.env.example` 和初始化脚本
- 内部测试完整的 Docker 部署流程

### 阶段 3：License 系统（2 周）
- 后端实现 License 生成、验证、续期 API
- twt-admin 增加 License 管理界面
- 实现离线验证降级逻辑

### 阶段 4：客户管理与运维（1-2 周）
- twt-admin 扩展客户管理功能（部署状态、版本追踪、服务连通性检查）
- 编写私有化部署文档和运维手册
- 编写升级/回滚脚本

### 阶段 5：首个客户试点（1-2 周）
- 选择一个客户进行试点部署
- 收集反馈，优化流程
