# TWT-Link 私有化部署方案

## Context

TWT-Link 是一套客服 SaaS 产品，当前以多租户模式运行在单台服务器上。业务模式是：我们提供软件服务，客户自备短信、邮箱、OSS 存储、域名等基础设施。现在需要设计私有化部署方案，让客户可以在自己的服务器上独立运行整套系统。

当前代码仓库是纯前端 monorepo（9 个 Vue 3 应用），后端服务不在此仓库中。已有的 `@twt/branding` 租户配置系统是很好的基础。

---

## 一、两种私有化模式

### 模式 A：轻量私有化（推荐起步）
- 客户使用独立域名，但前端仍部署在我们的基础设施上
- 通过现有的 hostname 租户解析机制，加载客户专属配置
- 后端 API 指向客户的独立数据库实例（数据隔离）
- 客户自备的 OSS/邮箱/短信 通过租户配置注入

适合：预算有限、不要求物理隔离的客户

### 模式 B：完全私有化
- 整套系统（前端 + 后端 + 数据库）部署在客户自己的服务器上
- 用 Docker Compose 打包，一键部署
- 客户完全掌控数据和基础设施

适合：对数据主权有严格要求的客户（金融、政府、大企业）

## 二、Docker 化部署架构

### 整体结构
```
docker-compose.yml
├── nginx          # 反向代理 + 静态文件服务（所有前端应用）
├── api-server     # 后端 API 服务
├── database       # PostgreSQL / MySQL
├── redis          # 缓存 + 会话
└── worker         # 异步任务（邮件发送、webhook 等）
```

### 前端打包策略
- 所有前端应用统一构建为一个 Nginx Docker 镜像
- 利用现有的 Vite base path 约定（`/<app-name>/`），所有应用共存于一个 Nginx 下
- 租户配置通过 `/tenants/{hostname}.json` 静态文件注入，与现有 `loadTenantConfig()` 机制完全兼容
- Nginx 配置模板化，启动时根据环境变量生成

### 需要新建的文件
```
deploy/
├── docker-compose.yml          # 编排文件
├── docker-compose.prod.yml     # 生产覆盖
├── nginx/
│   ├── Dockerfile              # 前端镜像
│   ├── nginx.conf.template     # Nginx 配置模板
│   └── entrypoint.sh           # 启动脚本（环境变量替换）
├── .env.example                # 环境变量模板
└── scripts/
    ├── init.sh                 # 初始化脚本
    ├── upgrade.sh              # 升级脚本
    └── backup.sh               # 备份脚本
```

## 三、配置注入方案

### 前端配置（客户可见）
沿用现有 `TenantConfig` 机制，扩展以下字段：

```typescript
// 在 packages/branding/src/types.ts 中扩展
interface TenantConfig {
  // ... 现有字段 ...
  
  // 新增：API 地址（私有化部署时指向客户自己的后端）
  apiBaseUrl: string
  wsBaseUrl: string        // WebSocket 地址
  
  // 新增：短信配置
  sms?: {
    provider: 'twilio' | 'aliyun' | 'tencent' | 'custom'
    // 具体凭证不放前端，仅标识 provider
  }
}
```

### 敏感配置（仅后端可见）
当前 `TenantConfig` 中的 `oss.accessKeyId/Secret` 暴露在前端 JSON 中，这是安全隐患。需要拆分：

- 前端配置：品牌、UI、域名、功能开关 → `/tenants/{hostname}.json`
- 后端配置：OSS 密钥、SMTP 密码、短信密钥、数据库连接 → 环境变量 / 后端配置文件

```env
# .env.example（私有化部署环境变量）
# === 基础 ===
DOMAIN=chat.customer.com
API_PORT=8080

# === 数据库 ===
DB_HOST=localhost
DB_PORT=5432
DB_NAME=twt_chat
DB_USER=twt
DB_PASSWORD=

# === OSS（客户自备）===
OSS_PROVIDER=aliyun          # aliyun | aws | qiniu
OSS_BUCKET=
OSS_REGION=
OSS_ACCESS_KEY_ID=
OSS_ACCESS_KEY_SECRET=

# === 邮箱（客户自备）===
SMTP_HOST=
SMTP_PORT=465
SMTP_USER=
SMTP_PASSWORD=
EMAIL_FROM_NAME=
EMAIL_FROM_ADDRESS=

# === 短信（客户自备）===
SMS_PROVIDER=twilio          # twilio | aliyun | tencent
SMS_ACCESS_KEY=
SMS_ACCESS_SECRET=
SMS_SIGN_NAME=

# === 许可证 ===
LICENSE_KEY=
```

## 四、客户管理体系

### 管理层级映射
```
twt-admin（我们的平台管理）
  └── 管理所有私有化客户
      ├── 客户 A（模式 A - 轻量私有化）
      ├── 客户 B（模式 B - 完全私有化）
      └── 客户 C（模式 A）
```

### 客户生命周期
1. **签约** → 在 twt-admin 创建客户记录，生成 License Key
2. **配置** → 收集客户的 OSS/邮箱/短信/域名信息
3. **部署** → 根据模式 A 或 B 执行部署
4. **验证** → 检查所有外部服务连通性
5. **交付** → 提供管理员账号和文档
6. **运维** → 版本升级、监控、续费

### 客户信息管理（twt-admin 扩展）
在现有 twt-admin 的代理商管理基础上，增加：
- 部署模式（轻量/完全私有化）
- 部署状态（待部署/运行中/已暂停/已下线）
- 当前版本号
- License 到期时间
- 外部服务配置状态（OSS/邮箱/短信 是否已配置且连通）
- 服务器信息（完全私有化模式下）

## 五、License 授权管理

### 方案
- 每个私有化客户分配唯一 License Key
- License 包含：客户 ID、到期时间、功能权限、坐席上限
- 后端启动时校验 License，定期（如每天）向我们的授权服务器验证
- 离线场景：License 内嵌签名，可离线验证基本有效性，但功能降级

### License 结构
```json
{
  "customerId": "customer-001",
  "plan": "professional",
  "maxAgents": 50,
  "features": ["ai-agent", "multi-channel", "analytics"],
  "expiresAt": "2027-04-22",
  "signature": "..."
}
```

## 六、软件升级策略

### 利用现有版本检测机制
项目已有 `vite-plugin-version` 生成 `version.json`，各应用的 `useVersionCheck` 会轮询检测新版本。这套机制在私有化部署中同样适用。

### 升级流程
**模式 A（轻量）：** 我们直接更新，客户无感知（与 SaaS 相同）

**模式 B（完全私有化）：**
1. 我们发布新版 Docker 镜像到私有 Registry
2. 通知客户有新版本可用
3. 客户执行升级脚本：
   ```bash
   ./scripts/upgrade.sh v2.5.0
   # 拉取新镜像 → 备份数据 → 数据库迁移 → 滚动更新
   ```
4. 支持回滚：`./scripts/upgrade.sh --rollback`

### 版本管理
- 语义化版本号（SemVer）
- 维护 CHANGELOG
- 重大版本提前通知，提供迁移指南
- 私有化客户可以选择跳过小版本，但大版本必须按序升级

## 七、安全注意事项

### 凭证安全
- **立即修复**：将 `TenantConfig` 中的 `oss.accessKeyId/Secret` 从前端配置中移除，改为后端环境变量
- 所有敏感信息通过环境变量注入，不进入代码仓库或前端可访问的文件
- Docker secrets 或 Vault 管理生产凭证

### 数据隔离
- 模式 A：数据库级隔离（每个客户独立 schema 或独立数据库实例）
- 模式 B：物理隔离（独立服务器，天然隔离）
- OSS 隔离：每个客户使用自己的 bucket，不存在交叉访问

### 网络安全
- 强制 HTTPS
- API 通信使用 JWT + 租户隔离中间件
- 私有化部署建议客户配置防火墙规则
- License 验证通信加密签名

### 代码安全
- 私有化交付的是编译后的 Docker 镜像，不交付源码
- 前端代码经过 Vite 构建压缩，但本质上前端代码客户可见，不应包含核心业务逻辑

## 八、迁移路径（分阶段）

### 阶段 1：配置安全化（1-2 周）
- 将 OSS/SMTP 敏感凭证从 `TenantConfig` 前端配置中剥离
- 后端增加配置 API，前端只获取非敏感的租户配置
- 修改 `packages/branding/src/types.ts`，移除敏感字段

### 阶段 2：Docker 化（2-3 周）
- 编写前端 Nginx Dockerfile（统一构建所有应用）
- 编写 docker-compose.yml（含后端、数据库、Redis）
- 编写 `.env.example` 和初始化脚本
- 内部测试完整的 Docker 部署流程

### 阶段 3：License 系统（2 周）
- 后端实现 License 生成、验证、续期 API
- twt-admin 增加 License 管理界面
- 实现离线验证降级逻辑

### 阶段 4：客户管理（1-2 周）
- twt-admin 扩展客户管理功能（部署状态、版本追踪、服务连通性检查）
- 编写私有化部署文档和运维手册
- 编写升级/回滚脚本

### 阶段 5：首个客户试点（1-2 周）
- 选择一个客户进行模式 A 试点
- 收集反馈，优化流程
- 然后推进模式 B 试点
