## 问题陈述

代理商要求白标功能，想用自己的域名和品牌销售 TWT Chat。具体需求：
- 所有前端（官网、客服端、访客端、控制台、后台）都要白标
- 完全隐藏 TWT 品牌
- 支付用 Stripe，平台收款后分成给代理商
- APP 用统一 APP + 工作空间切换

## 需求证据

**真实需求信号：**
- 有具体的代理商在谈（知道名字和背景）
- 谈到了钱（有付费意愿）
- 代理商用过产品，有具体反馈
- 代理商明确表达："想用自己的域名去卖"

**商业模式：**
- 代理商自主定价
- 按订单百分比分成
- 支付走 Stripe，平台收款后分成给代理商
- 注意：这个模式是我们设计的，还没跟代理商验证

## 现状

**代码库架构：**
- Vue 3 monorepo，5 个应用：web-agent（客服端）、web-admin（运营后台）、web-customer（访客端）、web-website（官网）、mobile-app（APP）
- 已有项目切换逻辑（web-agent 中的 currentProjectId）
- 完整的权限系统（13 个权限组，3 种角色）
- 功能门控系统（免费版/专业版，18 个受限功能，包括"隐藏品牌"）
- 设计令牌系统（CSS 变量，品牌颜色已抽象）
- 品牌元素硬编码在各应用中
- 无真实后端，所有数据都是模拟数据

**部署方式：**
- rsync 到远程预览服务器
- 预览地址：`https://pm.pro.jishu666.com/<app-name>/index.html`

## 约束条件

1. 所有前端都必须白标（不只是访客端）
2. 完全隐藏 TWT 品牌（不要 "Powered by TWT"）
3. **APP 每个代理商独立打包**（不是统一 APP + 工作空间切换）
4. 支付走 Stripe，平台收款后分成
5. 产品稳定性、运营能力、代理商筛选"都不是问题"（创始人判断）
6. **这是产品原型**，后端 API 无需真实实现，用静态 JSON 模拟即可
7. 数据隔离在原型阶段无需实现

## 前提

**前提 1：代理商真的需要"所有前端白标"**
创始人坚持所有前端都要白标。但 Codex 挑战了这个前提，认为这是执念，不是成交必要条件。真正需要白标的可能只是对外成交面（官网、访客端、登录页、支付页）。

**验证方式：** 拿 48 小时原型，只白标对外成交面，如果代理商愿意签 LOI 愿意付订金，说明范围被扩大了。

**创始人决定：** 坚持所有前端白标。

**前提 2：完全隐藏品牌会让 TWT 变成可替代的技术供应商**
如果完全隐藏品牌，代理商的客户不知道 TWT 存在，代理商随时可以换掉 TWT，因为客户认的是代理商品牌，不是 TWT。这与创始人"3 年后白标会更有价值"的判断矛盾。

**创始人决定：** 确认完全隐藏品牌。

**前提 3：商业模式还没验证**
"代理商自主定价 + 按订单百分比分成"这个模式是我们设计的，还没跟代理商确认。如果代理商不接受这个模式，技术方案可能需要调整。

**风险：** 技术实现后发现商业模式不可行。

## 跨模型视角

**Codex 独立审查（冷读）：**

**1) 最强版本**
不是做一个换 logo 的白标功能，是真正在做一个可分销的多租户客服平台底座：代理商拿你的产品，用自己的域名、品牌、销售体系去卖，但结算、产品迭代、核心控制权仍然在你这边。说白了，你想做的不是 TWT Chat 本体，而是 TWT 作为渠道基础设施。

**2) 最关键的一句**
"他们想要代理我们的TWT Chat，然后用自己的域名去卖"

这句把本质全说透了。你真正的客户不只是终端商家，而是渠道商；你该优先构建的不是 UI 白标，而是 **域名映射 + 租户隔离 + 品牌配置 + 结算归属 + 代理后台** 这套分销操作系统。谁还把这事理解成改前端皮肤，谁就是没看懂生意。

**3) 挑战的前提：所有前端都必须白标**
这大概率是创始人的执念，不一定是成交的必要条件。证明 Codex 对的证据很简单：拿一个 48 小时原型，只白标官网 + 访客端 + 登录页 + 支付页，客服工作台和后台先保留 TWT 内部痕迹，如果代理商依然愿意签 LOI 愿意付订金，说明你们之前在扩大范围，真正该先做的是对外成交面，不是内部全栈重涂。

**4) 48 小时原型建议**
- 做一个共享包 `packages/branding`，定义租户配置
- 所有 web 应用启动时按 window.location.host 解析租户
- 先打通 web-website、web-customer、web-admin，再把 web-agent 只改外壳层
- 预置两个 demo 租户，用两个预览域名演示同一套代码不同品牌
- 支付只做假通路或最小 Stripe Checkout
- 跳过：移动端真改造、每租户独立数据库隔离、自动分账、邮件模板、SEO、埋点、多语言

**一句话结论：** 先证明代理商会不会为"可卖"给钱，不要先证明你们能不能把全家桶都换皮。后者是体力活，前者才是生死线。

**我的看法：**
Codex 的核心洞察是对的：这不是 UI 换皮问题，是分销操作系统问题。但创始人坚持所有前端白标，我会在方案中体现这个决策，同时指出风险。

## 考虑的方案

### 方案 A：运行时多租户（推荐）

**核心思路：** 单一代码库，运行时根据域名动态加载租户配置。

**重要变更：** 基于新需求，APP 改为每个代理商独立打包，不再是统一 APP。

**架构：**

**实现步骤：**

**工作量：** XL（6-8 周，2-3 人）
- Web 白标：2 周

- TWT 管理后台：1 周
- APP 自动化打包：2 周

**风险：** 中高
- APP 独立打包的复杂度远超预期
- 每个代理商需要独立的开发者账号和证书
- 应用商店审核可能拒绝"马甲包"
- 更新和版本管理非常复杂

**优点：**
- 单一代码库，Web 端易于维护
- Web 端升级一次，所有租户同步
- 租户配置灵活，易于扩展

**缺点：**
- **APP 独立打包复杂度极高**
- 每个代理商需要独立的开发者账号（iOS $99/年，Android $25 一次性）
- 应用商店可能拒绝"马甲包"（多个相似 APP）
- APP 更新需要为每个代理商单独打包和上架
- 推送证书、签名证书管理复杂
- 如果有 10 个代理商，就需要维护 10 个 iOS APP + 10 个 Android APP

### 方案 B：构建时多租户

**核心思路：** 为每个代理商构建独立的前端包，部署到不同的域名。

**架构：**
```
packages/branding/
  ├── tenants/
  │   ├── default.json
  │   ├── reseller-a.json
  │   └── reseller-b.json
  └── build-script.ts       # 构建脚本，为每个租户生成独立包

dist/
  ├── default/              # TWT 默认版本
  ├── reseller-a/           # 代理商 A 版本
  └── reseller-b/           # 代理商 B 版本
```

**构建流程：**
1. 读取租户配置
2. 替换代码中的品牌占位符
3. 为每个租户生成独立的构建产物
4. 部署到对应的域名

**实现步骤：**
1. 创建构建脚本，支持多租户构建
2. 修改 `vite.config.ts`，支持环境变量注入
3. 替换所有硬编码的品牌元素为环境变量
4. 配置 CI/CD，自动为每个租户构建和部署
5. 实现代理商管理后台（触发构建和部署）

**工作量：** L（3-4 周，2 人）

**风险：** 中
- 构建和部署复杂度高
- 每次升级需要为所有租户重新构建
- 租户数量增加时，构建时间线性增长

**优点：**
- 每个租户完全独立，定制化能力强
- 无运行时性能开销
- 可以为不同租户使用不同的代码版本

**缺点：**
- 维护成本高，升级困难
- 构建和部署复杂
- 租户数量增加时，成本线性增长

### 方案 C：混合方案（最小可验证版本）

**核心思路：** 先做运行时多租户的最小版本，只白标对外成交面，验证商业模式后再扩展。

**第一阶段（2 周）：**
- 只白标：官网（web-website）、访客端（web-customer）、登录页、支付页
- 客服端（web-agent）、后台（web-admin）保留 TWT 品牌
- 实现基础的租户配置和域名映射
- 集成 Stripe Checkout（最小版本，不做自动分账）

**第二阶段（2 周）：**
- 如果代理商愿意签 LOI 或付订金，继续白标客服端和后台
- 实现代理商管理后台
- 实现 Stripe Connect 自动分账

**第三阶段（2 周）：**
- 实现 APP 工作空间切换
- 完善代理商运营功能（数据看板、财务结算）

**工作量：** S（第一阶段 2 周，1-2 人）

**风险：** 低
- 快速验证商业模式
- 避免过度投入

**优点：**
- 快速上线，验证需求
- 投入小，风险低
- 可以根据反馈调整方向

**缺点：**
- 不符合创始人"所有前端白标"的要求
- 可能需要向代理商解释为什么客服端还有 TWT 品牌

## 推荐方案

**推荐：方案 A（运行时多租户）+ 重大警告**

**理由：**
1. **符合创始人要求** - 可以实现所有前端白标 + 独立 APP
2. **Web 端技术成熟** - 行业标准做法，风险低
3. **代理商后台功能完整** - 包含所有 7 个功能模块

**但是，我必须指出严重的风险：**

### ⚠️ APP 独立打包的巨大成本

你选择了"每个代理商独立 APP"，这会带来：

**成本（每个代理商）：**
- Apple Developer 账号：$99/年
- Google Play 账号：$25 一次性
- 开发者时间：首次打包 2-3 天，每次更新 1 天
- 应用商店审核时间：iOS 1-3 天，Android 几小时到几天

**如果有 10 个代理商：**
- 年度账号成本：$990 + $250 = $1,240
- 维护 20 个 APP（10 个 iOS + 10 个 Android）
- 每次更新需要打包 20 次，上架 20 次
- 应用商店可能拒绝"马甲包"（4.3 Spam 条款）

**Codex 的建议是对的：统一 APP + 工作空间切换。**

但你坚持独立 APP，原因是"客服端登录也存在代理商 + 项目的选择问题"。

**我的建议：**
1. **短期（MVP）**：只做 Web 白标，暂不做 APP
2. **中期**：做统一 APP + 工作空间切换，验证需求
3. **长期**：如果代理商愿意承担成本（开发者账号 + 维护费用），再做独立 APP

**如果你坚持现在就做独立 APP，工作量会从 2-3 周变成 6-8 周，风险从低变成中高。**

**关键实现细节：**

**1. 租户配置加载**
```typescript
// packages/branding/src/loader.ts
export async function loadTenantConfig(): Promise<TenantConfig> {
  const hostname = window.location.hostname
  
  // 从后端 API 获取租户配置
  const response = await fetch(`/api/tenants/by-domain/${hostname}`)
  if (response.ok) {
    return response.json()
  }
  
  // 降级到默认配置
  return DEFAULT_TENANT_CONFIG
}
```

**2. Vue 3 上下文注入**
```typescript
// packages/branding/src/context.ts
import { inject, provide, InjectionKey } from 'vue'

const TENANT_KEY: InjectionKey<TenantConfig> = Symbol('tenant')

export function provideTenant(config: TenantConfig) {
  provide(TENANT_KEY, config)
}

export function useTenant() {
  const tenant = inject(TENANT_KEY)
  if (!tenant) {
    throw new Error('Tenant not provided')
  }
  return tenant
}
```

## 我注意到你的思考方式

1. **你对所有前提都很坚定** - 当我挑战"所有前端白标"和"完全隐藏品牌"时，你都坚持原计划。当 Codex 建议"统一 APP + 工作空间切换"时，你选择了"每个代理商独立 APP"。这显示出强烈的执行意愿，但也可能是执念。

2. **你说"都做"** - 当我问代理商后台的 7 个功能哪 3 个最关键时，你说"都做"。这要么是你们资源充足，要么是你低估了开发复杂度。7 个功能页面 + APP 独立打包，工作量是 6-8 周，不是 2-3 周。

3. **你选择了最复杂的方案** - 独立 APP 的成本是统一 APP 的 10 倍以上。如果有 10 个代理商，你需要维护 20 个 APP，每次更新需要打包 20 次。这是巨大的运营负担。

4. **商业模式还没验证** - "代理商自主定价 + 按订单百分比分成"是你们设计的，还没跟代理商确认。建议先验证商业模式和最小范围，再投入 6-8 周的开发。

5. **你认为 3 年后白标会更有价值，但选择了完全隐藏品牌** - 这两者矛盾。如果完全隐藏品牌，代理商随时可以换掉你们，你们没有品牌护城河。

**我的建议：**
- 先跟代理商确认他们是否真的需要"独立 APP"和"所有前端白标"
- 先做最小版本（Web 白标 + 代理商后台核心功能），验证需求
- 如果代理商愿意为独立 APP 付费（承担开发者账号 + 维护成本），再做独立 APP

**Codex 说得对：先证明代理商会不会为"可卖"给钱，不要先证明你们能不能把全家桶都换皮。后者是体力活，前者才是生死线。**
