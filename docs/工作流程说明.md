# TWT 项目工作流程说明

> 本文档描述 TWT 客服 SaaS 原型项目的完整开发流程，适用于产品、设计、开发团队协作。

## 一、项目概览

**目标**：通过组合已审核的组件生成可运行的原型页面，替代静态 Axure 线框图。

**技术栈**：
- 前端：Vue 3 + TypeScript + Vite
- 包管理：pnpm (monorepo)
- 主要应用：`web-agent`（客服工作台）

**预览地址**：https://pm.pro.jishu666.com/web-agent/index.html

**项目阶段**：原型验证阶段（MVP）

---

## 二、核心工作流程

### 流程总览

```
阶段 0: 需求收集 → 阶段 1: 需求讨论 → 阶段 2: PRD 生成 → 阶段 3: PRD 校验
                                                                    ↓
阶段 9: 产品验收 ← 阶段 8: 部署预览 ← 阶段 7: 提交代码 ← 阶段 6: 自测审查
      ↓ 通过                                                      ↑
阶段 10: 上线                                    阶段 4: 设计评审（可选）
      ↑ 不通过：返回阶段 5                              ↓
                                              阶段 5: 开发实现
```

**关键决策点**：
- 阶段 3 → 阶段 4：PRD 校验不通过，返回阶段 2 修改
- 阶段 4 → 阶段 5：设计评审不通过，返回设计师修改
- 阶段 9 → 阶段 10：验收不通过，返回阶段 5 修复

### 时间预估（SLA）

| 阶段 | 预期时长 | 可并行工作 | 备注 |
|------|---------|-----------|------|
| 阶段 0 | 1-2 天 | - | 复杂需求可能需要 1 周 |
| 阶段 1 | 2-4 小时 | - | 单个功能模块 |
| 阶段 2 | 30 分钟 | - | AI 辅助生成 |
| 阶段 3 | 15 分钟 | - | AI 自动校验 |
| 阶段 4 | 1-2 天 | 可与阶段 5 并行启动 | 设计师出图时开发可先搭架构 |
| 阶段 5 | 0.5-3 天 | - | 视功能复杂度 |
| 阶段 6 | 1-2 小时 | - | 自测 + 代码审查 |
| 阶段 7 | 10 分钟 | - | 提交代码 |
| 阶段 8 | 5 分钟 | - | 自动化部署 |
| 阶段 9 | 1-2 小时 | - | 产品验收 |

**单个功能完整周期**：3-5 个工作日（简单功能）/ 1-2 周（复杂功能）

**并行优化建议**：
- 设计师出图期间，开发可先搭建页面架构和路由
- 多个独立功能可分配给不同开发并行开发
- PRD 生成和校验可在需求讨论后立即完成，无需等待

### 风险管理

#### 常见风险与应对

| 风险类型 | 具体表现 | 应对措施 | 责任人 |
|---------|---------|---------|--------|
| **需求不清** | 讨论后仍有歧义 | 暂停开发，补充需求讨论会议 | 产品 |
| **技术难点** | 开发评估超过 3 天 | 技术预研 → 降级方案 → 需求调整 | 开发 |
| **设计延期** | 设计稿超期 2 天未交付 | 使用低保真原型先开发，后续调整 | 产品 |
| **频繁返工** | 同一功能验收失败 ≥3 次 | 召开复盘会，分析根因 | 产品+开发 |
| **依赖阻塞** | 等待后端接口超过 1 周 | 使用 Mock 数据先行开发 | 开发 |
| **技术债累积** | 代码质量下降，bug 增多 | 每月安排 1 周技术债偿还时间 | 开发 |

#### 质量门禁（必须停止的情况）

- 🚫 PRD 校验连续 3 次不通过 → 产品经理重新梳理需求
- 🚫 核心功能验收失败 → 禁止部署，必须修复

### 度量指标（KPI）

#### 效率指标
- **需求交付周期**：从阶段 0 到阶段 9 的平均时长（目标：≤1 周）
- **PRD 一次通过率**：阶段 3 校验一次通过的比例（目标：≥80%）
- **验收一次通过率**：阶段 9 一次通过的比例（目标：≥70%）
- **返工次数**：单个功能平均返工次数（目标：≤1 次）

#### 质量指标
- **线上 bug 率**：上线后 1 周内发现的 bug 数量（目标：≤2 个/功能）
- **决策文件覆盖率**：有决策文件支撑的 PRD 比例（目标：100%）
- **代码规范遵守率**：自测检查清单通过率（目标：100%）

#### 协作指标
- **需求变更率**：开发中需求变更的比例（目标：≤20%）
- **沟通效率**：决策记录响应时间（目标：讨论后 1 小时内记录）

### 会议与沟通机制

#### 定期会议

| 会议类型 | 频率 | 时长 | 参与者 | 目标 |
|---------|------|------|--------|------|
| **需求评审会** | 每周一次 | 1-2 小时 | 产品+开发+设计 | 评审本周待开发需求 |
| **站会（可选）** | 每日 | 15 分钟 | 全员 | 同步进度、识别阻塞 |
| **功能验收会** | 按需 | 30 分钟 | 产品+开发 | 集中验收已完成功能 |
| **月度复盘会** | 每月一次 | 1 小时 | 全员 | 回顾指标、优化流程 |

#### 异步沟通规范

- **决策记录**：讨论达成共识后 1 小时内记录到 `memory/`
- **需求变更**：通过飞书/钉钉通知相关方，并更新决策文件
- **技术问题**：开发遇阻超过 2 小时，主动同步产品寻求方案调整
- **进度同步**：每日下班前更新任务状态（进行中/已完成/受阻）

#### 冲突解决机制

1. **产品与开发意见不一致**：
   - 开发说明技术难点和成本
   - 产品评估业务价值和优先级
   - 共同寻找折中方案（降级/分期）
   - 无法达成一致时，升级至技术负责人或业务方

2. **设计与产品意见不一致**：
   - 设计说明设计理念和用户体验考量
   - 产品说明业务目标和功能约束
   - 优先保证功能完整性，视觉可迭代优化

### 流程优化建议

#### 针对小团队的优化策略

**当前痛点 → 优化方案**：

1. **人力不足，进度压力大**
   - 优先级严格管理：只做 P0/P1 需求
   - 技术债务可控：每月预留 20% 时间偿还
   - 复用优先：优先使用现有组件，减少重复开发

2. **需求频繁变更**
   - 需求冻结机制：开发启动后，非紧急需求延后到下个迭代
   - 变更成本评估：每次变更需评估影响范围和时间成本

3. **沟通成本高**
   - 决策文件作为单一事实来源，减少重复沟通
   - 异步优先：非紧急问题通过文档/飞书沟通，减少会议

4. **质量与速度平衡**
   - 原型阶段：功能完整性 > 代码质量 > 视觉完美度
   - 核心功能：必须有自测检查清单
   - 边缘功能：可快速实现，后续迭代优化

### 阶段 1：需求讨论与决策记录

**参与角色**：产品经理、开发、设计、AI 助手

**前置条件**：
- 已完成阶段 0 的需求筛选
- 明确本次讨论的功能模块范围

**讨论内容**：
1. **业务规则**：权限判断、状态流转、数据联动
2. **交互细节**：点击、hover、展开/折叠、弹窗触发时机
3. **校验规则**：字段格式、必填项、边界条件
4. **异常处理**：错误提示文案、兜底方案、降级策略
5. **技术约束**：性能要求、浏览器兼容性、第三方依赖

**决策记录流程**：
1. 在对话中充分讨论上述内容
2. 当某个规则达成共识时，**立即记录到决策文件**
   - 触发词：用户说「记一下」「记录决策」
   - 存储位置：`memory/decisions-*.md`
   - 文件命名：`decisions-功能模块名.md`（如 `decisions-Email渠道.md`）
3. AI 助手自动生成结构化决策文档

**决策文件示例结构**：
```markdown
---
name: Email 渠道核心决策
description: 邮件会话、引用处理、线程识别、多发件人规则
type: project
---

## 邮件会话重新开启
- 规则：访客回复已关闭会话的邮件时，自动创建新会话
- 原因：避免历史会话污染，保持会话独立性

## 邮件引用处理
- 显示：仅展示最近一条引用内容
- 折叠：超过 3 行自动折叠，点击「展开」查看全文
```

### 阶段 2：PRD 生成

**参与角色**：产品经理 + AI 助手

**前置条件**：
- 必须先完成阶段 1 的决策记录
- 相关决策文件已存在于 `memory/` 目录

**执行命令**：
```bash
/prd
```

**AI 工作流程**：
1. 读取 `memory/` 中相关的决策文件
2. 结合当前对话内容生成 PRD
3. 输出符合规范的 PRD 文档

**PRD 编写规范**（参考 `memory/feedback_prd_writing.md`）：
- ❌ 禁止：UI 样式描述（颜色、字号、间距等）
- ✅ 必须包含：
  - 校验规则（字段格式、必填项、边界条件）
  - 交互逻辑（点击、hover、展开/折叠）
  - 业务规则（权限判断、状态流转、数据联动）
  - 异常处理（错误提示、兜底方案）

### 阶段 3：PRD 校验

**参与角色**：产品经理

**执行命令**：
```bash
/prd-review
```

**校验内容**：
- 是否包含 UI 样式描述（不合规）
- 是否缺少校验规则、交互逻辑、业务规则
- 是否符合已有决策文件的约定
- 是否遵循全局 UI 规范（Toast 文案、弹窗样式等）

**输出**：
- 合规项列表
- 不合规项列表 + 修改建议

### 阶段 5：开发实现

**参与角色**：前端开发 + AI 助手

**开发规范**（参考 `CLAUDE.md`）：

#### 4.1 依赖约束
- `web-agent` 页面**只能**导入 `@twt/ui-agent` 和 `@twt/tokens`
- 禁止跨端导入（如 `ui-admin`、`ui-customer`）

#### 4.2 样式规范
- **禁止**在页面/视图文件中写硬编码颜色或间距
- **必须**使用 `--agent-*` CSS 变量（来自 `@twt/tokens/agent.css`）
- 示例：
  ```css
  /* ❌ 错误 */
  .my-button { background: #3b82f6; padding: 12px; }

  /* ✅ 正确 */
  .my-button {
    background: var(--agent-color-brand-primary);
    padding: var(--agent-space-12);
  }
  ```

#### 4.3 组件使用
- 优先使用 `@twt/ui-agent` 中的现有组件
- 组件类名遵循 BEM 风格，前缀 `agent-`
- 布局使用 `AgentAppShell` 组件的插槽系统

#### 5.4 常用命令
```bash
pnpm install              # 安装依赖
pnpm dev:web-agent        # 启动开发服务器
pnpm build:web-agent      # 生产构建
```

## 四、团队协作要点

### 4.1 产品经理
1. **需求阶段**：明确目标用户、使用场景、优先级
2. **讨论阶段**：主动说「记一下」触发决策记录
3. **PRD 阶段**：生成前确认相关决策文件已存在，使用 `/prd-review` 自查质量
4. **验收阶段**：对照 PRD 和决策文件逐项验收

### 4.2 开发人员
1. **开发前**：先读 `CLAUDE.md` 和相关 `memory/decisions-*.md`
2. **开发中**：严格遵守依赖约束和样式规范
3. **提交前**：完成自测检查清单，使用 `/commit` 生成规范 commit message
4. **部署后**：在预览环境验证功能正常

### 4.3 设计师
1. 设计稿需标注使用的 token 变量名（如 `--agent-color-brand-primary`）
2. 新增设计 token 需同步更新 `packages/tokens/`
3. 参与需求讨论，明确交互细节和异常场景处理

### 4.4 跨角色协作建议
- **异步沟通**：决策文件作为单一事实来源，避免口头约定
- **及时同步**：重大决策变更需通知相关方并更新决策文件
- **问题追溯**：遇到争议时回溯决策文件，查看当时的决策依据

## 五、常见问题

### Q1：如何查看已有的决策记录？
```bash
ls memory/decisions-*.md
cat memory/MEMORY.md  # 查看索引
```

### Q2：PRD 校验不通过怎么办？
根据 `/prd-review` 输出的不合规项逐条修改，重点检查：
- 是否包含 UI 样式描述
- 是否缺少校验规则和业务逻辑

### Q3：如何添加新的设计 token？
1. 编辑 `packages/tokens/src/agent.json`
2. 运行构建命令生成 CSS 文件
3. 在组件中使用 `var(--agent-*)`

### Q4：部署失败怎么排查？
1. 检查 `apps/upcode.sh` 脚本权限
2. 确认 `apps/chanpin6.p` 密钥文件存在
3. 查看 rsync 输出日志

### Q5：需求讨论时什么内容需要记录到决策文件？
需要记录的：
- ✅ 业务规则（权限判断、状态流转）
- ✅ 交互逻辑（点击后的行为、弹窗触发条件）
- ✅ 校验规则（字段格式、必填项）
- ✅ 异常处理（错误提示文案、降级方案）

不需要记录的：
- ❌ UI 样式（颜色、字号、间距）→ 由设计 token 管理
- ❌ 技术实现细节（用什么库、怎么写代码）→ 开发自行决策
- ❌ 临时讨论内容（未达成共识的想法）

### Q6：决策文件和 PRD 的区别是什么？
- **决策文件**：跨会话持久化的业务规则，可被多个 PRD 引用
- **PRD**：单次功能开发的完整需求文档，引用决策文件中的规则

示例：`decisions-Email渠道.md` 记录邮件会话的通用规则，多个邮件相关 PRD 都会引用这些规则。

### Q7：验收不通过后如何处理？
1. 产品经理记录问题清单（功能缺失、交互不符、bug 等）
2. 开发根据问题清单修复
3. 修复后重新部署预览环境
4. 产品经理再次验收，直到通过

### Q8：如何处理需求变更？
1. 评估变更影响范围（是否影响已有决策）
2. 如影响决策文件，先更新决策文件
3. 如影响 PRD，更新 PRD 并重新校验
4. 通知相关开发人员变更内容
5. 开发根据新 PRD 调整代码

## 六、工具迁移指南

如需将此工作流迁移到其他 AI 编程工具（Cursor、Windsurf、Codex 等）：

### 6.1 需要迁移的文件
- `CLAUDE.md` → 重命名为目标工具的配置文件（如 `.cursorrules`）
- `memory/` 目录 → 复制到项目根目录或目标工具的上下文目录
- `~/.claude/CLAUDE.md` → 复制到目标工具的全局配置

### 6.2 迁移脚本示例
```bash
# 备份当前配置
cp -r memory ~/Desktop/twt-memory-backup

# 迁移到 Cursor
cp CLAUDE.md .cursorrules
echo "\n请在每次对话时读取 memory/ 目录下的决策文件" >> .cursorrules
```
