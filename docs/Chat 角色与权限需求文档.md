# Chat 角色与权限需求文档



## 一、简介

### 1、目的

在 Chat 项目内引入项目级角色与权限体系，解决当前“成员能看到什么菜单、能操作什么功能”不可配置的问题，满足管理员与普通客服的职责分离。

典型场景：

1. 管理员可以查看所有功能菜单，对所有功能拥有操作权限。
2. 客服可以查看部分功能菜单，对部分功能拥有操作权限。例如：客服可以看客服列表，但不能进行邀请、编辑、启停用、删除、重置密码、分配角色等管理操作。
3. 同一成员在不同项目中可拥有不同角色，进入不同项目时按当前项目权限加载菜单和按钮。

### 2、范围

本期范围：

1. 新增项目级角色体系，包含系统角色与自定义角色。
2. 新增角色管理页面，支持查看、新增、编辑、删除自定义角色及配置权限。
3. 团队成员管理接入角色能力，支持邀请时指定角色、成员详情修改角色，并将邀请记录合并到客服列表统一展示。
4. 覆盖 Chat Web 端现有一级菜单和关键功能按钮的权限控制。
5. 登录成功和切换项目时，系统需按当前项目加载对应的角色与权限结果。
6. 前端按权限隐藏菜单与按钮，后端接口强校验。

本期不包含：

1. 不单独设计数据范围权限。
2. 不改造已有“删除成员”业务语义，仅新增权限门禁。
3. 不支持一个成员在同一项目下绑定多个角色。
4. 不支持复制其他自定义角色，仅支持从客服模板创建。

## 二、角色描述

| 角色 | 类型 | 说明 |
| --- | --- | --- |
| 管理员 | 系统角色 | 项目级最高管理角色，默认拥有全部权限 |
| 客服 | 系统角色 | 接待型角色，可处理会话和查看必要数据，无管理类权限 |
| 自定义角色 | 自定义角色 | 由管理员创建，可自定义菜单权限与功能权限 |

补充规则：

1. 项目创建者在创建项目时自动成为该项目的管理员。
2. 项目创建者永久不可降级、不可删除。
3. 系统角色 `管理员`、`客服` 固定存在，不可编辑、不可删除。
4. 同一成员在不同项目中的角色互不影响。

## 三、产品概述

### 1、权限控制维度

| 维度 | 说明 |
| --- | --- |
| 菜单权限 | 控制一级菜单、二级菜单是否可见 |
| 功能权限 | 控制菜单下关键按钮和操作是否可见 |
| 数据权限 | 本期不拆分数据范围，沿用现有项目隔离规则 |

### 2、角色模型

1. 系统角色固定为 `管理员`、`客服`。
2. 系统角色仅支持查看权限定义，不支持修改角色名、不支持修改权限、不支持删除。
3. 支持新增自定义角色。
4. 自定义角色默认复制 `客服` 角色权限模板。
5. 自定义角色支持修改角色名称、调整权限、删除角色。
6. 若角色已绑定成员，不允许删除，需先完成成员改绑。

### 3、成员绑定规则

1. 成员在单个项目下只能绑定 1 个角色。
2. 邀请成员时必须带出角色，默认值为 `客服`。
3. 管理员可在成员详情中调整非创建者成员角色。
4. 若成员在当前项目无有效角色，不允许进入该项目，并提示：`账号无权限，无法登录该项目`。

## 四、名词解释

| 名称 | 说明 |
| --- | --- |
| 系统角色 | 系统预置角色，固定存在，不可编辑、不可删除 |
| 自定义角色 | 管理员新增的角色，可配置权限 |
| 菜单权限 | 页面入口权限，决定菜单或子菜单是否展示 |
| 功能权限 | 页面内操作权限，决定按钮和关键操作是否展示 |
| 项目创建者 | 创建当前项目的成员，默认且永久为管理员 |
| 邀请记录 | 已发出邀请但尚未激活为正式客服的记录，默认仅管理员可见 |

## 五、全局规则

### 1、权限生效规则

1. 页面可见：必须具备对应页面的查看权限。
2. 按钮可见：必须同时具备页面可见权限和对应操作权限。
3. 无权限表现：菜单、二级菜单、按钮直接隐藏，不使用置灰。
4. 具体操作执行时，后端必须校验成员是否具备对应权限，前端不可作为唯一权限校验。
5. 权限变更后，后端立即按最新权限生效；前端在刷新页面、重新进入页面或触发后续请求时按最新权限更新展示。

### 2、项目隔离规则

1. 权限结算维度固定为“成员 + 项目”。
2. 同一邮箱加入多个项目时，每个项目单独绑定角色。
3. 成员切换项目后，按当前项目角色重新加载菜单与按钮。
4. 项目 A 的角色与权限变更不影响项目 B。

### 3、兼容规则

1. 存量项目需自动补齐 `管理员`、`客服` 两个系统角色。
2. 存量项目创建者回填为管理员，其他存量成员回填为客服。
3. 现有成员状态、删除状态、历史会话数据、报表归属逻辑保持不变。

## 六、功能特性

### 1、设置新增“角色”菜单

#### 1.1 入口

1. 在客服页面中新增 `角色` tab。
2. 仅具备角色查看权限的成员可见。
3. 管理员默认可见，客服默认隐藏。

#### 1.2 角色列表

列表字段如下：

| 字段 | 说明 |
| --- | --- |
| 角色名称 | 展示角色名 |
| 角色类型 | 系统角色 / 自定义角色 |
| 权限概览 | 展示已配置的一级菜单和关键功能摘要 |
| 绑定成员数 | 当前项目下使用该角色的成员数 |
| 创建人 | 系统角色显示 `系统`，自定义角色显示创建人 |
| 创建时间 | 系统角色显示初始化时间，自定义角色显示创建时间 |

交互规则：

1. 系统角色显示 `系统角色` 标识。
2. 系统角色仅支持查看，不显示编辑、删除按钮。
3. 自定义角色显示编辑、删除按钮。
4. 有角色新增权限时显示 `新增角色` 按钮。
5. 点击角色名称或查看按钮进入角色详情页。

#### 1.3 角色详情

1. 角色详情页展示角色基础信息、权限树、绑定成员数。
2. 系统角色详情页为只读态。
3. 自定义角色详情页在具备角色编辑权限和角色权限配置权限时可进入编辑。

#### 1.4 新增角色

字段定义：

| 字段 | 说明 |
| --- | --- |
| 角色名称 | 必填，项目内唯一，不能与 `管理员`、`客服` 重名 |
| 权限配置 | 展示权限树，支持勾选菜单权限和功能权限 |

默认值：

1. 角色名称默认为空。
2. 权限默认复制 `客服` 角色权限模板。

结果规则：

1. 保存成功后角色立即加入角色列表。
2. 新角色创建后可立即用于成员分配。

#### 1.5 编辑角色

1. 仅自定义角色支持编辑。
2. 支持修改角色名称与权限配置。
3. 保存后后端按最新权限立即生效。
4. 已绑定该角色的成员在后续请求中按最新权限校验。

#### 1.6 删除角色

1. 仅自定义角色支持删除。
2. 删除前需二次确认。
3. 若角色仍有关联成员，禁止删除，并提示先调整成员角色。
4. 删除成功后角色从列表移除。

### 2、团队成员管理接入角色

#### 2.1 团队-客服列表

1. `设置 > 团队 > 客服` 页面改为由客服列表查看权限控制。
2. 页面保持单列表结构，不新增单独的“邀请记录”页面或 tab。
3. 列表同时承载两类记录：
   1. 已激活客服
   2. 邀请记录
4. 普通客服默认可见只读，但仅可看到“已激活客服”记录，不可看到邀请记录。
5. 具备邀请记录查看权限的成员，才可在同一列表中看到邀请记录。
6. 无权限时隐藏入口。

页面结构规则：

1. 顶部保留当前页面的坐席统计区和右上角 `邀请客服` 按钮。
2. 坐席统计区仅对具备邀请记录查看权限或邀请客服权限的成员展示，普通客服默认不展示，避免从统计口径反推出隐藏的邀请记录。
3. 列表字段沿用现有页面结构，统一调整为：

| 字段 | 已激活客服 | 邀请记录 |
| --- | --- | --- |
| ID | 展示成员 ID | 展示 `-` |
| 姓名 | 展示姓名和头像 | 展示 `-` |
| 昵称 | 展示角色标签 + 昵称 | 展示邀请角色标签 + `待激活` |
| 创建/邀请时间 | 展示创建时间 | 展示最近邀请时间 |
| 邮箱 | 展示邮箱 | 展示邮箱 |
| 会话限制 | 展示会话限制 | 展示 `-` |
| 在线状态 | 展示在线/离开/离线 | 展示 `-` |
| 状态 | 展示正常/禁用 | 展示待激活/已过期 |
| 操作 | 展示成员操作 | 展示邀请操作 |

记录展示规则：

1. 同一列表内默认按“创建/邀请时间”倒序展示。
2. 已激活客服和邀请记录统一参与分页。
3. 普通客服请求该页面时，后端仅返回已激活客服记录。
4. 管理员请求该页面时，后端返回已激活客服和邀请记录。

坐席统计口径：

1. 当前服务包含坐席总数 = 套餐总坐席数。
2. 使用中 = 已激活客服数量 + 状态为“待激活”的邀请记录数量。
3. 剩余 = 当前服务包含坐席总数 - 使用中。
4. 状态为“已过期”的邀请记录不占用坐席。
5. 删除邀请后立即释放坐席。
6. 重新邀请不重复占位，只刷新原记录的有效期和最近邀请时间。

#### 2.2 邀请成员

1. 邀请流程新增 `角色` 字段，默认选中 `客服`。
2. 仅具备邀请客服权限时显示 `邀请客服` 按钮。
3. 角色下拉展示当前项目所有可用角色。
4. 发送邀请前校验当前剩余坐席数必须大于 0，否则不允许发送邀请。
5. 同一项目下，同一邮箱只允许存在 1 条未完成邀请记录。
6. 若该邮箱已有状态为“待激活”或“已过期”的邀请记录，不新增第二条记录，而是在原记录上执行重新邀请逻辑。
7. 被邀请成员加入项目后，自动绑定邀请时选择的角色，并将邀请记录转为正式成员数据。

邀请记录生成规则：

1. 邀请发送成功后，在客服列表中生成 1 条邀请记录。
2. 初始状态为 `待激活`。
3. 邀请链接有效期为 24 小时。
4. 超过 24 小时未激活，邀请记录状态自动变为 `已过期`。

重复发送与重新邀请规则：

1. 点击 `重新邀请` 后，旧邀请链接立即失效。
2. 系统生成新的邀请链接，并将有效期重新计算为 24 小时。
3. 同一邀请记录支持重复重新邀请，但需要限频。
4. 限频规则：
   1. 前端按钮进入 60 秒冷却。
   2. 后端按 `项目 + 邮箱` 做 60 秒限流。
5. 60 秒内重复触发时，返回提示：`请稍后再试`。

#### 2.3 成员详情/编辑

1. 成员详情新增 `角色` 字段。
2. 仅具备成员角色分配权限时，角色字段可编辑。
3. 管理员不可修改项目创建者角色。
4. 若目标成员已被删除或禁用，角色字段仅回显，不允许再分配。

#### 2.4 邀请记录操作边界

1. 邀请记录未激活前，不视为正式客服成员。
2. 邀请记录不允许执行以下正式成员操作：
   1. 发起聊天
   2. 账号详情
   3. 修改密码
   4. 启用/停用账号
   5. 删除成员
3. 邀请记录仅允许执行以下操作：
   1. 重新邀请
   2. 删除邀请
4. 删除邀请后：
   1. 该记录从列表中移除
   2. 该邀请对应的所有历史链接立即失效
   3. 通过历史邀请链接进入时，提示邀请已失效或已撤销

#### 2.5 成员操作权限

| 权限能力 | 说明 |
| --- | --- |
| 编辑成员 | 编辑正式成员基本信息 |
| 启用/停用成员 | 启用或停用正式成员 |
| 删除成员 | 沿用现有删除成员逻辑，仅新增权限门禁 |
| 重置密码 | 重置正式成员密码 |
| 分配角色 | 修改正式成员角色 |
| 查看邀请记录 | 查看邀请记录 |
| 重新邀请 | 重新发送邀请 |
| 删除邀请 | 删除邀请 |

规则：

1. 无对应权限时，相关按钮直接隐藏。
2. 普通客服默认无邀请记录查看、重新邀请、删除邀请权限。
3. 普通客服可以查看正式客服列表，但不能看到邀请记录，也不能重新邀请任何人。
4. 成员只读场景下，仅保留查看能力，不展示任何管理按钮。

### 3、系统角色默认权限

#### 3.1 管理员

管理员默认拥有全部权限。

#### 3.2 客服

客服默认采用“接待只读增强”模板：

| 模块 | 权限结果 |
| --- | --- |
| 会话 | 可见；允许转移会话、添加客服，仍受现有会话规则限制 |
| 访客 | 可见 |
| 档案 | 可见 |
| 个人资料/通知设置 | 可见且可维护 |
| 快捷回复 | 可见；可维护个人回复；可查看和使用公共回复；不可维护公共回复 |
| 客服页面 | 可见只读，仅可看到正式客服列表 |
| 邀请记录 | 隐藏 |
| 角色 | 隐藏 |
| 会话设置 | 隐藏 |
| 安装 | 隐藏 |
| 报表 | 隐藏 |
| 营销 | 隐藏 |

### 4、自定义角色配置规则

1. 自定义角色可自由勾选菜单权限与功能权限。
2. 自定义角色不支持多角色叠加。
3. 成员最终权限以当前项目绑定的单一角色为准。

### 5、登录与项目切换

1. 登录成功后，系统按项目维度加载成员角色和权限集。
2. 多项目成员进入不同项目时，按当前项目独立加载菜单与按钮。
3. 若当前项目下账号被删除、被禁用或无角色，则该项目不可进入。
4. 登录成功和切换项目时，系统需识别当前项目角色、项目创建者身份以及当前成员可见功能范围。

## 七、数据变更

### 1、数据实体

新增或补充以下实体能力：

| 实体 | 字段/能力 |
| --- | --- |
| 项目角色表 | 角色名、角色类型、角色权限配置、创建人、创建时间、是否系统角色、是否可编辑、是否可删除 |
| 项目成员角色关系 | 项目 ID、成员 ID、角色 ID |
| 邀请记录表 | 项目 ID、邮箱、邀请角色 ID、邀请人、首次邀请时间、最近邀请时间、过期时间、状态、是否占用坐席 |


## 八、迁移方案

### 1、存量项目初始化

1. 对所有存量项目自动补齐 `管理员`、`客服` 两个系统角色。
2. 系统角色在同一项目下只允许创建一次，不可重复生成。

### 2、存量成员回填

1. 项目创建者回填为 `管理员`。
2. 其他存量成员回填为 `客服`。
3. 已删除、已禁用成员保留历史状态，仅补齐角色关系，供历史数据展示与权限判断使用。

### 3、邀请记录初始化

1. 现有存量邀请数据若存在，需迁移为邀请记录并挂载到统一的客服列表查询结果中。
2. 迁移后的邀请记录需至少补齐邮箱、邀请角色、邀请人、最近邀请时间、过期时间、状态。
3. 已过期邀请迁移后状态为 `已过期`，且不占用坐席。

### 4、兼容要求

1. 存量会话、报表、营销、成员删除逻辑不被改写。
2. 历史数据归属保持原有规则，不因角色能力上线发生变化。

## 九、假设与默认规则

1. 本期只做角色与权限控制，不做数据范围权限。
2. “删除成员”沿用现有逻辑，本 PRD 不重新定义删除语义。
3. 自定义角色首期默认从客服模板创建，不支持从其他角色复制。
4. 单项目单成员仅允许 1 个角色，不支持多角色并集。
5. 无权限表现统一为“直接隐藏”，不是置灰。
