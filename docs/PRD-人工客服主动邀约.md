# 人工客服主动邀约需求文档

## 一、 需求背景与目标

*   **核心目标**：在现有访客小组件基础上增加「人工客服主动邀约」能力。当客服主动向访客发送消息时，通过气泡通知强化展示，打破"等客上门"的被动局面，提升客服主动触达效率和会话转化率。

*   **业务定位**：强化客服的主动服务能力。通过高优先级的真人邀约气泡机制，确保客服的主动沟通能够及时、醒目地传达给访客，提升访客响应率。

## 二、 人工客服主动邀约功能

### 2.1 触发时机

客服在 Web 端主动向访客**发送会话消息**时（访客/客户列表、群发消息），且访客未展开聊天面板。

**注意**：单聊/群聊暂不触发提醒。

### 2.2 展示形态

在访客端小组件图标上方，弹出带有**消息文本**的通知小气泡。气泡右上角常驻 `[X]` 关闭按钮。

### 2.3 交互规则

1.  **叠加与顶替**：最多纵向堆叠展示 **3 条**最新消息。收到第 4 条时，最旧的一条自动消失，整体上移。

2.  **消失与兜底机制**：气泡弹出后，不会主动消失。只能访客手动点 X 关闭气泡，客服原生图标上显示【未读会话数量红点】兜底。

3.  **点击动作**：访客点击任一消息气泡，立即展开完整的聊天主界面。同时关闭通知气泡

### 2.4 多客服主动邀约处理

**场景描述**：访客端当前已经接收并展示了客服 A 的主动邀约气泡，此时客服 B 也向该访客发送了主动邀约消息。

**气泡展示逻辑（先入为主）**：

1.  前端增加气泡展示状态判断。

2.  当已有气泡处于展示状态时，对于新到达的其他客服邀约消息，前端在**气泡展示层面直接拦截并忽略**，不更新气泡内容。只在客服图标显示未读红点。

3.  直到当前气泡的生命周期结束（如访客点击查看、手动关闭），才允许展示下一条新到达的邀约气泡。

### 2.5 异常场景处理

1.  **网络断开场景**：访客网络断开期间，客服发送的邀约消息暂存在服务端。访客重新连接后，按时间顺序依次展示气泡（遵循最多 3 条堆叠规则）。

3.  **页面刷新/重新加载**：访客刷新页面后，未读的邀约消息重新展示气泡（遵循最多 3 条堆叠规则）。已手动关闭的气泡不再重复展示。

4.  **超时处理**：邀约消息发送后，如果 30 秒内未收到访客端确认（网络问题、页面未加载等），服务端标记为"未送达"，客服端显示发送状态提示。

### 2.6 移动端适配

1.  **气泡尺寸**：移动端气泡宽度自适应屏幕宽度（左右各留 16px 边距），高度根据消息内容自适应，单条气泡最大高度 120px，超出部分显示省略号。

2.  **堆叠展示**：移动端同样最多堆叠 3 条气泡。在小屏设备（如 iPhone SE，屏幕高度 568px）上，3 条气泡总高度约 300px，不会遮挡主要内容区域。

3.  **触摸交互**：移动端点击气泡区域展开聊天面板，点击 `[X]` 关闭按钮关闭气泡。关闭按钮触摸区域不小于 44x44px，符合移动端可点击区域标准。

## 三、 数据埋点需求

为验证功能效果，需要采集以下数据：

### 3.1 核心指标

| 埋点事件 | 触发时机 | 必传参数 |
| --- | --- | --- |
| `agent_invite_sent` | 客服发送主动邀约消息 | 客服 ID、访客 ID、消息内容长度、时间戳 |
| `agent_invite_shown` | 访客端展示邀约气泡 | 访客 ID、气泡位置（第 1/2/3 条）、时间戳 |
| `agent_invite_clicked` | 访客点击邀约气泡 | 访客 ID、停留时长（从展示到点击）、时间戳 |
| `agent_invite_closed` | 访客手动关闭邀约气泡 | 访客 ID、停留时长（从展示到关闭）、时间戳 |
| `agent_invite_replaced` | 营销气泡被邀约气泡替换 | 访客 ID、被替换的营销任务 ID、时间戳 |

### 3.2 成功标准

功能上线后，通过以下指标评估效果：

1.  **访客响应率**：点击邀约气泡的访客数 / 收到邀约气泡的访客数 ≥ 30%
2.  **会话转化率**：点击后发起会话的访客数 / 点击邀约气泡的访客数 ≥ 60%
3.  **主动邀约占比**：客服主动发起的会话数 / 总会话数 ≥ 15%
4.  **平均响应时长**：从气泡展示到访客点击的平均时长 ≤ 60 秒

## 四、 防冲突与全局免打扰

为保障访客体验，防止过度打扰，系统需在前端判定弹窗资格。

### 4.1 与主动营销功能的冲突处理

**背景说明**：主动营销（Campaigns）功能已上线，支持通过预设规则自动弹出营销气泡。人工客服主动邀约需要与已上线的营销功能进行冲突处理。

#### 4.1.1 营销弹窗内部免打扰

当判定满足 Campaigns 触发条件（身份、页面、时间）时，需校验以下红线，命中任一条则**严格阻断弹窗**：

1.  访客当前的聊天窗口正处于「展开/开启」状态。

2.  本次访问页面的生命周期内，该访客已手动点击过此营销气泡的关闭 `[X]` 按钮。即展示过营销弹窗后不再弹出其他营销卡片。

#### 4.1.2 人工消息绝对优先权

"真实客服交互"的优先级永远大于"系统自动营销"。

1.  **强制替换**：若当前屏幕正展示着「自动化营销气泡」，此时突然收到客服的「人工主动邀约消息」，前端需**立刻销毁营销气泡**，替换弹出人工消息气泡。

2.  **全局免疫**：只要客服主动发送过消息，在当前页面生命周期内，该访客对所有的自动化营销规则免疫，不再触发任何营销弹窗。

### 4.2 主动营销卡片展示冲突处理

**触发场景**：系统设定的主动营销卡片正处于「延迟弹出」的倒计时阶段。

**用户行为**：访客在此倒计时期间内，主动点击了客服图标并展开了聊天窗口。

**系统处理逻辑（阻断销毁）**：

*   系统监听到聊天面板被主动打开的事件后，立即**清除/销毁**当前正在等待的所有营销卡片展示任务。

*   即使访客后续关闭了聊天面板，之前未展示的营销卡片也**不再弹出**。

## 五、 人工邀约与营销功能交互流程图

### 核心逻辑节点解析

1.  **营销气泡自身防冲突（阻断销毁）**： 在营销气泡还在"准备中"（倒计时）的时候，一旦访客主动打开了聊天窗口，系统判定访客已有主动沟通意愿，直接**清除所有等待的任务**。

2.  **人工消息的绝对优先级（强制替换）**： 无论营销气泡展示得多好，只要"真实的客服"发话了，系统会**立刻销毁营销气泡**，给人工消息让路。并且在此后当前页面的访问周期内，该访客不再受任何系统自动营销打扰。

3.  **多客服邀约防冲突（先入为主）**： 处理了多个客服同时抢夺一个访客注意力的问题。一旦客服 A 的气泡已经展示，客服 B 的消息只能以"小红点"形式低调提示，必须等访客处理完当前气泡（点击或关闭），才能进入下一轮展示。
