# 产品经理交接文档

## 一、需求流程

### 1.1 需求上线前

**原型 & 设计评审**
- 需求开发前，需完成产品原型评审 → 设计稿评审 → 研发需求宣贯（三步缺一不可）
- 需求宣贯前必须确保：需求已通过评审 + 需求文档已撰写完毕 + 设计稿已通过评审
- 如果无需设计介入或产品介入的需求可直接跳过设计稿评审或产品原型评审阶段

**验收**
- 需求上线前，若产品与设计人员参与了前期需求设计，需由产品与设计共同完成验收。验收重点核查：交互是否符合需求定义、功能逻辑是否正确
- 发现问题的处理方式：
  - Bug → 反馈给测试同事，或直接录入 GTD 缺陷中，并将缺陷负责人指定为对应需求负责人
  - 设计问题 → 反馈给 UI 同事
  - 其他调整 → 直接反馈给对应开发同事

### 1.2 上线前通知

在钉钉「上线确认及同步群」发布当天上线内容，格式示例：

```
【发布说明】：
版本号
- [需求ID] 需求名称 @负责人 @相关人员
- [需求ID] 需求名称 @负责人 @相关人员

@测试同事（必须@）
```

### 1.2 上线前国际化

**国际化翻译**：若开发告知「xxx 端需要翻译」，登录 [Crowdin](https://crowdin.com) 找到对应项目完成翻译

### 1.3 需求上线

1. **更新日志**：
   - 评估客服端本次上线内容，若存在新增功能、体验优化等需告知用户的变更，应在 **Chat IDS 后台**的「更新日志」模块中进行新增发布。
   - 如果有官网帮助中心/常见问题等内容修改 → 在 **TWT IDS 后台**中更新
2. **服务群公告**：评估是否在 TWT 服务群发布更新说明
   - 新功能或优化功能 → 一般需要发布
   - 若该功能来源于服务群中用户反馈 → 需在群里发布更新说明

## 二、GTD 需求管理

### 2.1 录入需求

新建需求时需填写以下信息：
- **标签**：添加对应端口的标签（如客服端、访客端、APP 等）
- **负责人**：一般为产品自己；若需求已明确指定给开发或设计，则指定给对应同事
- **提出人**：需求的提出人

**优先级判断**：综合考虑提出人、需求重要性等因素确定。

### 2.2 需求状态流转

```
待确认 → 待评审 → 待排期 → 开发排期 → ...
```

| 状态 | 含义 | 操作方 |
|------|------|--------|
| 待确认 | 需求已录入，产品需出方案并组织评审 | 产品 |
| 待评审 | 产品方案已确认，需设计参与出设计稿 | 产品变更状态，设计跟进 |
| 待排期 | 设计稿通过评审，等待开发排期 | 产品变更状态，开发自行排期 |

> 若需求非常明确、不需要产品参与，可直接将状态设为「待排期」，负责人指定给对应开发同事。

### 2.3 需求原型 & 文档完成后

- 将原型链接和需求文档链接粘贴到 GTD 对应需求的**评论**中，并 @ 相关同事（设计、开发）
- 若需求有设计参与，需在该需求的**子工作项**中为设计新增一条工作项，负责人指定为对应设计同事

## 三、需求排期节奏

| 周期 | 时间 | 内容 |
|------|------|------|
| 周排期 | 一般每周三或周四 | 确认下周需求范围和优先级 |
| 月排期 | 每月月底 | 确认下月需求范围和优先级 |

**GTD 迭代命名规则**：`2026.4.5` = 2026 年 4 月第 5 周的需求

## 三、多端迭代说明

| 端 | 迭代方式 | 说明 |
|----|----------|------|
| 客服端 / 访客端 / 官网 | 按需求迭代 | 随需求上线，无版本号 |
| APP（移动端） | 按版本迭代 | 如 1.7.0、1.7.1，版本节奏慢于网页端 |

**注意**：网页端新增功能时，需评估是否影响 APP 端。若有影响（如新消息类型、功能逻辑变更），需尽快安排 APP 版本跟进支持。

### 4.5 新增项目时的配置步骤

1. 在 `apps/` 下创建新项目目录，添加 `package.json`（`name` 字段与目录名一致）
2. 在根目录 `package.json` 的 `scripts` 中添加对应的 `dev:`、`build:`、`deploy:` 命令
3. 若需要部署，确认 `apps/upcode.sh` 支持该项目名（通常自动兼容）
4. 运行 `pnpm install` 使 workspace 识别新包

## 五、重点需求背景说明

### 5.1 接入 Email 渠道

**背景**：支持客服通过邮件渠道与访客沟通，统一在客服工作台收发邮件。

**核心方案**：
- 支持 Gmail OAuth 授权连接邮箱
- 邮件会话以聊天气泡形式展示，每封邮件为一条消息
- **后端剥离引用内容**：收到邮件时只保存新回复，移除历史引用（`<blockquote>`、`> ` 引用行等）
- 线程识别依赖邮件头（`Message-ID` / `In-Reply-To` / `References`），不依赖主题
- 支持多发件人：客服可选择不同已连接邮箱回复，默认选中该会话最初接收邮件的邮箱
- 自动过滤 no-reply 类邮件（noreply@、do-not-reply@ 等），不在消息队列展示
- 删除邮箱时同步撤销 OAuth token

### 5.2 Webhooks 优化 — 支持自定义渠道

**背景**：原有 Webhook 功能较基础，本次优化支持配置多个渠道（飞书、钉钉、Slack、企业微信、自定义），并支持自定义推送字段。

**核心方案**：
- 每个 Webhook 渠道对应独立 URL
- 同一 URL 不允许重复添加，保存时校验唯一性
- 支持自定义展示字段（访客姓名、消息内容、服务客服名称等 11 个字段可选），至少勾选 1 项
- 推送消息格式：每个字段自动换行，提升可读性

### 5.3 功能按服务版本区分（免费版 / 专业版）

**背景**：产品分层收费，部分高级功能仅专业版可用，需在前端做版本门控。

**核心方案**：
- 两个版本：**免费版** 和 **专业版**
- 导航菜单不锁定，用户可正常进入任何页面
- 免费版或服务过期后点击受限功能时弹「升级到专业版」弹窗
- 专业版功能共 18 项，包括：Copilot 智能回复、文本润色、聊天翻译、Webhooks、AI Agent、群发消息、黑名单等
- 免费版可查看部分内容（如知识库、角色列表），但新增/编辑/删除操作锁定
- 版本升级后需刷新页面生效

## 七、上线发布说明模板

版本号规则：`上线日期 + 当日序号`，例如 `20260420001`（2026-04-20 当天第 1 次上线），同日多次上线则依次为 `0002`、`0003`。

需求 ID 填写本次上线关联的 GTD 需求 ID，多个用顿号分隔。

各端分开撰写，无变更的端不写。

```
【发布说明】
版本号：20260420001
需求ID：98323、98217、97335

Chat 客服端：
【新增】xxx
【优化】xxx
【修复】xxx

Chat 访客端：
【新增】xxx
【优化】xxx
【修复】xxx
```
