## 二、需澄清问题清单

### Q1：列表默认排序规则未明确

**问题描述**：
PRD 中未说明会话列表和访客列表的默认排序字段和排序方向。

**原因分析**：
- 会话列表有「发起时间」和「接待时间」两个时间字段，均支持排序
- 访客列表有「首次访问」和「最后访问」两个时间字段，均支持排序
- 未明确页面初次加载时按哪个字段排序、升序还是降序

**影响范围**：
- 影响前端开发：不确定默认排序逻辑
- 影响测试用例：无法验证默认排序是否正确
- 影响用户体验：不同开发人员可能实现不同的默认排序

**建议补充的规则**：
- 会话列表默认按「发起时间」倒序排列（最新的在前）
- 访客列表默认按「最后访问」倒序排列（最近访问的在前）
- 「来源渠道」列不支持用户手动排序（仅展示和筛选）

**风险等级**：中等

---

### Q2：筛选操作的交互反馈未定义

**问题描述**：
PRD 中未说明点击「搜索」按钮后是否有 loading 状态，以及筛选失败时的错误提示。

**原因分析**：
- 筛选操作需要向后端发起请求
- 请求过程中用户可能重复点击或不清楚系统是否在处理
- 请求失败时用户无法得知失败原因

**影响范围**：
- 影响用户体验：无 loading 状态可能导致用户重复点击
- 影响异常处理：筛选失败时无明确提示

**建议补充的规则**：
- 点击「搜索」按钮后，按钮显示 loading 状态（禁用 + 加载图标）
- 筛选成功后，表格刷新展示结果，按钮恢复正常状态
- 筛选失败时，显示 Toast 提示：「加载失败，请重试」

**风险等级**：轻微

---

### Q3：历史数据兼容策略需确认

**问题描述**：
PRD 第 6 节「约束与依赖」中提到"本次上线前的历史会话和访客数据需补充 sourceChannel 字段"，但未明确具体的兼容策略。

**原因分析**：
- 历史数据的 sourceChannel 字段可能为 null 或不存在
- 未明确是通过数据迁移脚本补全，还是在展示层做兜底处理
- 未明确兜底显示值是「--」、「未知」还是其他

**影响范围**：
- 影响数据迁移方案：是否需要编写迁移脚本
- 影响前端开发：是否需要在展示层做 null 值处理
- 影响测试用例：需要准备历史数据进行兼容性测试

**建议补充的规则**：
- 明确历史数据处理策略：
  - 方案 A：数据迁移脚本将所有历史数据的 sourceChannel 设置为默认值（如「web」）
  - 方案 B：前端展示层对 null 值做兜底处理，显示「--」或「未知」
- 明确筛选时历史数据的处理：
  - 选择「全部」时，历史数据（sourceChannel 为 null）是否显示
  - 选择具体渠道时，历史数据是否被过滤掉

**风险等级**：中等
