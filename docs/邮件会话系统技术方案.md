# 邮件会话系统技术方案

## 一、邮件线程归类(Thread Management)

### 核心概念
邮件系统通过"线程ID"把同一主题的往来邮件串联起来,类似微信群聊的消息归组。

### 技术实现

#### 1. 线程识别规则
- 使用邮件头部的 `In-Reply-To` 和 `References` 字段
- 这两个字段记录了"这封邮件是回复哪封邮件的"
- 系统收到邮件时,检查这些字段,找到对应的会话

#### 2. 会话创建逻辑
```
收到新邮件 →
├─ 有 In-Reply-To/References?
│  ├─ 是 → 查找对应会话 → 追加到该会话
│  └─ 否 → 创建新会话
└─ 生成唯一会话ID(session_id)
```

#### 3. 数据库设计
- `email_sessions` 表:存储会话基本信息(主题、参与人、创建时间)
- `email_messages` 表:存储每封邮件,关联 session_id
- `email_attachments` 表:存储附件元信息
- 通过 `message_id`(邮件唯一标识)关联线程

#### 4. 产品层面体现
- 用户看到的是"会话列表",每个会话包含多条消息
- 类似微信聊天记录,按时间顺序展示
- 会话列表显示:最新消息预览、未读数、最后更新时间

## 二、发送附件(Outbound Attachments)

### 用户操作流程
```
客服在输入框点击附件按钮 →
选择文件(PDF/文档/文件等) →
文件上传到服务器 →
点击发送 →
邮件带附件发出
```

### 技术实现步骤

#### 1. 前端上传
- 用户选择文件后,立即上传到服务器临时存储
- 返回文件URL和元信息(文件名、大小、类型)
- 在输入框下方显示附件预览卡片
- 支持拖拽上传、点击上传两种方式

#### 2. 邮件发送
- 调用后端API,传递:消息文本 + 附件URL列表
- 后端从临时存储读取文件
- 使用 SMTP 协议发送邮件,附件编码为 MIME 格式
- MIME 是邮件附件的标准格式,所有邮箱都支持

#### 3. 数据存储
- 附件信息存入 `email_attachments` 表
- 关联 message_id,记录文件路径、大小、类型、上传时间
- 临时文件转为永久存储

#### 4. 产品层面体现
- 输入框支持拖拽上传、点击上传
- 显示上传进度条
- 附件卡片显示:文件图标、文件名、大小、删除按钮
- 发送前可删除已上传的附件

## 三、接收附件(Inbound Attachments)

### 技术实现

#### 1. 邮件解析
- 后端通过 IMAP/POP3 协议收取邮件
- 解析 MIME 格式,提取附件数据
- 保存到服务器文件存储(如 OSS/S3)

#### 2. 附件存储
- 生成唯一文件名(避免冲突)
- 数据库记录文件元信息:原始文件名、存储路径、大小、MIME类型

#### 3. 前端展示
- 消息气泡下方显示附件卡片
- 卡片内容:文件图标 + 文件名 + 大小
- 点击卡片触发预览

#### 4. 产品层面体现
```
┌─────────────────────────┐
│ 客户消息文本...           │
├─────────────────────────┤
│ 📎 产品需求文档.pdf       │
│    1.2 MB               │
│                         │
└─────────────────────────┘
```

## 四、附件下载

### 技术实现

#### 1. 安全验证
- 用户点击下载 → 后端验证权限(是否有权访问该会话)
- 生成临时下载链接(带签名,有效期10分钟)
- 防止未授权访问

#### 2. 下载方式
- **方式A**:直接返回文件流,浏览器触发下载
- **方式B**:返回预签名URL,前端跳转下载(推荐,减轻服务器压力)

#### 3. 预览支持
- 图片(jpg/png/gif):支持在线预览(新窗口打开或弹窗)
- PDF:支持在线预览
- 其他文件:直接下载

#### 4. 产品层面体现
- 点击附件卡片 → 浏览器弹出下载框
- 图片类附件可点击放大预览
- 下载进度提示(可选)

## 五、完整技术流程图

### 收邮件流程
```
IMAP服务器 → 后端定时拉取(轮询/Webhook) → 解析MIME →
提取附件 → 存储文件 → 写入数据库 → WebSocket推送 →
前端显示新消息
```

### 发邮件流程
```
前端上传附件 → 临时存储 → 客服点击发送 →
后端组装邮件(文本+附件) → SMTP发送 →
记录到数据库 → 前端显示已发送
```

### 线程管理流程
```
每封邮件携带 Message-ID → 回复时带上 In-Reply-To →
后端根据这些ID关联会话 → 前端按会话分组展示
```

## 六、需要的技术组件

### 后端
- **SMTP客户端**:发送邮件(如 Nodemailer)
- **IMAP客户端**:接收邮件(如 node-imap)
- **MIME解析器**:处理附件(如 mailparser)
- **文件存储**:OSS/S3 或本地存储
- **数据库**:存储会话、消息、附件元信息
- **WebSocket**:实时推送新消息

### 前端
- 文件上传组件(支持拖拽)
- 附件预览组件
- 下载触发逻辑
- WebSocket客户端(接收新消息推送)

## 七、关键技术点说明

### 1. Message-ID 的作用
- 每封邮件都有唯一的 Message-ID(如 `<abc123@example.com>`)
- 回复邮件时,会在 `In-Reply-To` 字段填入原邮件的 Message-ID
- `References` 字段记录整个线程的所有 Message-ID
- 通过这些字段,系统可以准确还原邮件对话关系

### 2. MIME 格式
- MIME(Multipurpose Internet Mail Extensions)是邮件附件的标准编码格式
- 支持多部分消息(文本 + 附件)
- 附件使用 Base64 编码传输
- 所有邮件客户端都支持 MIME 格式

### 3. SMTP vs IMAP
- **SMTP**(Simple Mail Transfer Protocol):发送邮件协议
- **IMAP**(Internet Message Access Protocol):接收邮件协议
- 系统需要同时支持两种协议

### 4. 实时推送方案
- 后端定时轮询 IMAP 服务器(如每30秒一次)
- 收到新邮件后,通过 WebSocket 推送给前端
- 前端实时显示新消息,无需刷新页面
