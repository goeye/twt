## 一、邮件协议机制

### 1.1 核心协议

| 协议 | 用途 | 说明 |
|------|------|------|
| **SMTP** | 发送邮件 | Simple Mail Transfer Protocol，负责邮件投递 |
| **IMAP** | 接收邮件 | Internet Message Access Protocol，支持服务端存储、多端同步 |
| **MIME** | 邮件编码 | Multipurpose Internet Mail Extensions，定义邮件内容和附件的编码格式 |

### 1.2 邮件头关键字段

每封邮件都包含标准头信息（RFC 5322），以下字段对会话系统至关重要：

| 字段 | 格式 | 作用 |
|------|------|------|
| `Message-ID` | `<unique-id@domain.com>` | 每封邮件的全局唯一标识符 |
| `In-Reply-To` | `<parent-message-id@domain.com>` | 指向父邮件的 Message-ID |
| `References` | 空格分隔的 Message-ID 链 | 完整的邮件线程引用链 |
| `Subject` | 文本 | 邮件主题，回复时自动添加 `Re:` 前缀 |
| `From` | 邮箱地址 | 发件人 |
| `To` / `CC` / `BCC` | 邮箱地址列表 | 收件人 / 抄送 / 密送 |
| `Content-Type` | MIME 类型 | 邮件内容格式（text/plain、text/html、multipart/mixed 等） |

**头信息示例：**

```
Message-ID: <msg003@customer.com>
In-Reply-To: <msg002@support.com>
References: <msg001@customer.com> <msg002@support.com>
Subject: Re: Enterprise Plan Quote Request
From: michael.brown@acme.com
To: support@company.com
Content-Type: multipart/mixed; boundary="----=_Part_12345"
```

### 1.3 MIME 格式

MIME 定义了邮件的多部分结构，一封包含附件的邮件结构如下：

```
multipart/mixed
├── multipart/alternative
│   ├── text/plain          ← 纯文本正文
│   └── text/html           ← HTML 正文
├── image/png (inline)      ← 内联图片
├── application/pdf         ← PDF 附件
└── application/zip         ← 其他附件
```

- 附件使用 Base64 编码传输
- 内联图片通过 `Content-ID` 在 HTML 中引用
- 所有主流邮件客户端兼容 MIME 格式

## 二、邮件线程识别（Email Threading）

### 2.1 线程识别机制

邮件线程识别**依赖邮件头信息**，不依赖 Subject 或发件人地址。

**识别优先级：**

1. **`In-Reply-To` + `References`**（最可靠）：通过 Message-ID 链精确匹配父邮件和完整线程
2. **`Subject` 匹配**（辅助）：去掉 `Re:` / `Fwd:` 前缀后匹配，仅作为补充手段
3. **发件人/收件人组合**（最低优先级）：同一组参与人 + 相近时间窗口，仅作兜底

**关键结论：**

- 线程识别**不依赖发件人地址** — 换邮箱回复仍在同一线程
- Subject 主题行是辅助手段，不是主要依据
- 只要 `In-Reply-To` 和 `References` 正确，线程就能准确匹配

### 2.2 会话匹配流程

```
收到新邮件
│
├─ 检查 In-Reply-To / References 字段
│  ├─ 有值 → 用 Message-ID 查找已有会话
│  │  ├─ 找到 → 追加到该会话
│  │  └─ 未找到 → 用 Subject + 参与人匹配
│  │     ├─ 找到 → 追加到该会话
│  │     └─ 未找到 → 创建新会话
│  └─ 无值（全新邮件） → 创建新会话
│
├─ 提取并存储 Message-ID（用于后续线程匹配）
├─ 解析邮件内容（正文 + 附件 + 引用内容）
└─ 推送到前端展示
```

### 2.3 换邮箱回复的线程保持

客服使用不同邮箱回复同一会话时，后端发送邮件**必须正确设置邮件头**：

```python
# 伪代码
reply_email = {
    'From': selected_from_address,         # 可以是任何已连接的邮箱
    'To': customer_email,
    'Subject': original_subject,           # 保持原主题（或加 Re:）
    'In-Reply-To': original_message_id,    # 关键！指向父邮件
    'References': original_references + ' ' + original_message_id,  # 关键！完整引用链
    'Message-ID': generate_unique_id(),    # 生成新的唯一 ID
    'Body': new_content
}
```

只要 `In-Reply-To` 和 `References` 正确设置，客户的邮件客户端会将回复归入同一会话。

## 三、引用内容处理（Quoted Content）

### 3.1 引用内容的产生机制

邮件不同于聊天消息 — 每封邮件都是**独立的完整文档**。回复邮件时，邮件客户端会自动将原始邮件内容引用到回复中（RFC 2822 标准行为）。

**引用格式：**

| 邮件格式 | 引用方式 | 示例 |
|----------|---------|------|
| 纯文本 | `>` 前缀 | `> 原始消息内容` |
| HTML | `<blockquote>` 标签 | `<blockquote>原始消息内容</blockquote>` |
| 归属行 | `On ... wrote:` | `On Wed, Mar 27, 2026 at 2:34 PM John <a@b.com> wrote:` |
| 分隔线 | 固定文本 | `-----Original Message-----` |

**雪球效应：**

如果不处理引用内容，邮件体积会随往来次数指数增长：

```
第 1 封：100 字
第 2 封：50 字新内容 + 100 字引用 = 150 字
第 3 封：30 字新内容 + 150 字引用 = 180 字
...
第 10 封：可能有几千字，但只有 20 字是新内容
```

### 3.2 后端引用内容剥离（生产环境）

**这是行业标准做法** — Intercom、Zendesk、Freshdesk 等主流客服系统均在后端剥离引用内容。

#### 剥离算法

```
收到邮件
│
├─ 解析 MIME，提取 HTML/纯文本正文
│
├─ 步骤 1：提取 <blockquote> 标签
│  ├─ 找到 → 提取为引用内容，从正文中移除
│  └─ 未找到 → 进入步骤 2
│
├─ 步骤 2：检测归属行（Attribution Line）
│  ├─ 匹配模式：
│  │  - /On .+? wrote:/i          （英文）
│  │  - /在 .+? 写道：/            （中文）
│  │  - /-----Original Message-----/
│  │  - /_{10,}/                   （下划线分隔线）
│  ├─ 找到 → 归属行及其后内容提取为引用，之前内容为正文
│  └─ 未找到 → 进入步骤 3
│
├─ 步骤 3：检测纯文本引用（> 前缀行）
│  ├─ 连续的 > 开头行 → 提取为引用内容
│  └─ 未找到 → 整封邮件视为新内容
│
└─ 输出：{ body: 新内容, quoted: 引用内容, raw: 原始邮件 }
```

#### Node.js 实现参考

```javascript
import { JSDOM } from 'jsdom';

function stripQuotedContent(htmlBody) {
  const dom = new JSDOM(htmlBody);
  const doc = dom.window.document;
  let quotedHtml = '';

  // 方法 1：提取 blockquote 标签
  const blockquotes = doc.querySelectorAll('blockquote');
  if (blockquotes.length > 0) {
    quotedHtml = Array.from(blockquotes).map(bq => bq.outerHTML).join('');
    blockquotes.forEach(bq => bq.remove());
    return {
      body: doc.body.innerHTML.trim(),
      quoted: quotedHtml
    };
  }

  // 方法 2：检测归属行
  const bodyHtml = doc.body.innerHTML;
  const match = bodyHtml.match(
    /(On .+? wrote:|在 .+? 写道：|-----Original Message-----|_{10,})/i
  );
  if (match && match.index !== undefined) {
    return {
      body: bodyHtml.substring(0, match.index).trim(),
      quoted: bodyHtml.substring(match.index).trim()
    };
  }

  return { body: bodyHtml, quoted: '' };
}
```

#### Python 实现参考

```python
from bs4 import BeautifulSoup
import re

def strip_quoted_content(html_body):
    """剥离邮件引用内容，返回 (新内容, 引用内容)"""
    soup = BeautifulSoup(html_body, 'html.parser')

    # 方法 1：提取 blockquote
    blockquotes = soup.find_all('blockquote')
    if blockquotes:
        quoted = ''.join(str(bq) for bq in blockquotes)
        for bq in blockquotes:
            bq.decompose()
        return str(soup), quoted

    # 方法 2：检测归属行
    text = str(soup)
    patterns = [
        r'On .+? wrote:',
        r'在 .+? 写道：',
        r'-----Original Message-----',
        r'_{10,}'
    ]
    for pattern in patterns:
        match = re.search(pattern, text, re.IGNORECASE)
        if match:
            return text[:match.start()].strip(), text[match.start():].strip()

    return text, ''
```

### 3.3 数据存储策略

```
email_messages 表:
├── id               : 消息 ID
├── session_id       : 关联会话
├── message_id       : 邮件 Message-ID
├── body             : 剥离引用后的新内容（前端展示用）
├── quoted_content   : 引用内容（展开查看用）
├── raw_content      : 原始完整邮件（调试/合规存档用）
├── content_type     : text/plain 或 text/html
├── from_email       : 发件人
├── to_email         : 收件人
├── subject          : 邮件主题
└── created_at       : 收到时间
```

**存储三份内容的原因：**
- `body`：前端默认展示，干净简洁
- `quoted_content`：用户点击「...」展开时展示
- `raw_content`：调试问题、合规审计时使用

### 3.4 前端展示方案（原型阶段）

由于原型阶段后端尚未实现引用剥离，前端临时模拟此逻辑。

#### 前端引用提取函数

```typescript
function extractQuotedContent(html: string): { body: string; quoted: string } {
  const doc = new DOMParser().parseFromString(html, 'text/html');
  let quotedHtml = '';

  // 方法 1：提取 blockquote 标签
  const blockquotes = doc.querySelectorAll('blockquote');
  if (blockquotes.length > 0) {
    quotedHtml = Array.from(blockquotes).map(bq => bq.outerHTML).join('');
    blockquotes.forEach(bq => bq.remove());
    // 移除归属行（On ... wrote:）
    const bodyHtml = doc.body.innerHTML;
    const cleaned = bodyHtml.replace(/On\s.+?\swrote:\s*<br\s*\/?>/gi, '')
                            .replace(/在\s.+?\s写道：\s*<br\s*\/?>/gi, '');
    return { body: cleaned.trim(), quoted: quotedHtml };
  }

  // 方法 2：检测归属行
  const bodyHtml = doc.body.innerHTML;
  const match = bodyHtml.match(
    /(On .+? wrote:|在 .+? 写道：|-----Original Message-----)/i
  );
  if (match && match.index !== undefined) {
    return {
      body: bodyHtml.substring(0, match.index).trim(),
      quoted: bodyHtml.substring(match.index).trim()
    };
  }

  return { body: html, quoted: '' };
}
```

## 四、邮件收发完整流程

### 4.1 收邮件流程

```
IMAP 服务器
│
├─ 后端定时轮询 / Webhook 通知
│
├─ 拉取新邮件
│
├─ 解析 MIME 格式
│  ├─ 提取邮件头（Message-ID, In-Reply-To, References, Subject, From, To）
│  ├─ 提取正文（text/plain + text/html）
│  ├─ 提取内联图片（Content-ID 关联）
│  └─ 提取附件（文件名、大小、MIME 类型、数据）
│
├─ 静默过滤
│  └─ 匹配 no-reply 类地址 → 丢弃，不进入消息队列
│     （noreply@、no-reply@、no_reply@、donotreply@、
│       do-not-reply@、do_not_reply@、mailer-daemon@）
│
├─ 线程匹配
│  ├─ 用 In-Reply-To / References 匹配已有会话
│  └─ 未匹配 → 创建新会话
│
├─ 引用内容剥离
│  ├─ 提取 blockquote / 归属行 / > 引用行
│  └─ 分离为 body + quoted_content
│
├─ 数据存储
│  ├─ 写入 email_messages（body + quoted + raw）
│  ├─ 写入 email_attachments（文件元信息）
│  └─ 附件上传到 OSS/S3
│
└─ 实时推送
   └─ WebSocket → 前端显示新消息
```

### 4.2 发邮件流程

```
客服编辑回复内容
│
├─ 选择发件人邮箱（支持多邮箱切换）
│
├─ 上传附件（可选）
│  ├─ 前端上传到临时存储
│  └─ 返回文件 URL + 元信息
│
├─ 点击发送
│
├─ 后端组装邮件
│  ├─ 设置邮件头
│  │  ├─ From: 客服选择的邮箱
│  │  ├─ To: 客户邮箱
│  │  ├─ Subject: 保持原主题（加 Re:）
│  │  ├─ Message-ID: 生成新的唯一 ID
│  │  ├─ In-Reply-To: 父邮件的 Message-ID    ← 关键
│  │  └─ References: 完整的 Message-ID 链     ← 关键
│  │
│  ├─ 组装 MIME 内容
│  │  ├─ text/html（正文）
│  │  └─ 附件（Base64 编码）
│  │
│  └─ 通过 SMTP 发送
│
├─ 数据存储
│  ├─ 写入 email_messages
│  └─ 临时附件转为永久存储
│
└─ 前端显示已发送消息
```

## 五、多发件人支持

### 5.1 设计方案

参考 Tidio 的实现方式，支持客服在回复时动态选择发件人：

- 支持多个已连接邮箱，客服可选择不同邮箱回复
- 默认邮箱：默认选中该会话最初接收邮件的邮箱
- 选择不持久化：每次进入会话重置为默认邮箱

### 5.2 邮箱连接要求

每个发件人邮箱需要完成以下配置：

| 配置项 | 说明 | 必要性 |
|--------|------|--------|
| SMTP 认证 | 用于发送邮件 | 必须 |
| IMAP 认证 | 用于接收邮件 | 必须 |
| SPF 记录 | DNS 记录，声明授权发件服务器 | 推荐 |
| DKIM 签名 | DNS 记录，邮件数字签名 | 推荐 |
| DMARC 策略 | DNS 记录，邮件认证策略 | 推荐 |

SPF、DKIM、DMARC 三项配置可以显著提高邮件送达率，避免被标记为垃圾邮件。

## 六、附件处理

### 6.1 接收附件

```
解析 MIME → 识别附件部分 → 读取 Base64 数据 → 解码 → 存储到 OSS/S3
```

**存储信息：**
- 原始文件名、唯一存储路径、文件大小、MIME 类型
- 关联 message_id，记录到 `email_attachments` 表

### 6.2 发送附件

```
前端选择文件 → 上传到临时存储 → 发送时读取文件 → Base64 编码 → 组装 MIME → SMTP 发送
```

**支持方式：** 拖拽上传、点击上传

### 6.3 附件预览与下载

| 类型 | 处理方式 |
|------|---------|
| 图片（jpg/png/gif） | 在线预览（点击放大） |
| PDF | 在线预览 |
| 其他文件 | 直接下载 |

**安全策略：**
- 下载前验证用户权限（是否有权访问该会话）
- 生成临时预签名 URL（有效期 10 分钟）

## 九、行业对标

### 主流客服系统的邮件处理方式

所有主流系统均采用**「聊天式展示 + 后端剥离引用」**架构：

| 平台 | 引用内容处理 | 线程识别 | 多发件人 |
|------|-------------|---------|---------|
| **Intercom** | 后端剥离，残留用左侧竖线标记 | Message-ID 线程识别 | 支持，需 DNS 认证 + 权限配置 |
| **Zendesk** | 简化邮件线程（2022 后默认） | References + In-Reply-To | 需第三方应用支持 |
| **Freshdesk** | 默认折叠引用，提供自动移除功能 | 标准邮件头匹配 | 支持多邮箱 |
| **Tidio** | 「...」展开查看引用 | 标准邮件头匹配 | 回复时动态选择发件人 |
| **Help Scout** | 按时间排序，区分角色 | 标准邮件头匹配 | 支持多邮箱 |

### 我们的方案定位

- 引用内容处理：参考 Freshdesk/Tidio，后端剥离 + 前端可展开
- 线程识别：遵循 RFC 标准，Message-ID / In-Reply-To / References
- 多发件人：参考 Tidio，动态选择发件人
