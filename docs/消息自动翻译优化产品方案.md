# 消息自动翻译优化产品方案

## 一、当前问题

用户在会话中时，新消息会自动翻译。但用户离开会话后收到的消息，再次进入会话时不会自动翻译，需要手动点击翻译按钮，体验差。

## 二、优化目标

用户进入会话时，所有未翻译的消息自动开始翻译，无需手动操作。

---

## 三、钉钉/飞书的实际做法

### 核心逻辑

**触发时机：用户进入会话时**

1. 用户打开会话
2. 系统检查哪些消息还没有翻译
3. 自动开始翻译这些消息
4. 消息先显示"翻译中..."
5. 翻译完成后自动更新为翻译结果

### 关键特点

- 不是后台提前翻译好，而是用户进入会话时才触发
- 翻译结果会保存，下次进入直接显示，不会重复翻译
- 即使有很多未读消息，也会全部自动翻译

---

## 四、翻译开关逻辑

### 两级开关设计

**1. Copilot 全局开关（系统级）**
- 位置：Copilot 设置中
- 作用：控制整个系统的翻译功能
- 优先级：最高

**2. 客服会话个人开关（会话级）**
- 位置：每个客服会话中
- 作用：控制当前会话是否翻译
- 优先级：次于全局开关

### 开关判断逻辑

```
是否翻译 = 全局开关 AND 会话开关

全局开关关闭 → 不翻译（无论会话开关状态）
全局开关开启 + 会话开关关闭 → 不翻译
全局开关开启 + 会话开关开启 → 翻译
```

### 开关状态表

| 全局开关 | 会话开关 | 是否翻译 |
|---------|---------|---------|
| 关闭 | 关闭 | ❌ 不翻译 |
| 关闭 | 开启 | ❌ 不翻译 |
| 开启 | 关闭 | ❌ 不翻译 |
| 开启 | 开启 | ✅ 翻译 |

## 五、产品方案

### 场景 1：客服在会话中（保持现有逻辑）

```
新消息到达 → 检查开关 → 自动翻译 → 显示翻译结果
```

### 场景 2：客服不在会话中（新增逻辑）

```
客服离开会话
  → 有新消息到达
  → 暂不翻译

客服再次进入会话
  → 检查全局开关和会话开关
  → 如果都开启，检查未翻译的消息
  → 自动开始翻译
  → 显示"翻译中..."
  → 翻译完成后显示结果
```

### 场景 3：客服再次进入已翻译过的会话

```
客服进入会话
  → 消息已有翻译结果
  → 直接显示翻译内容（无需等待）
```

### 场景 4：开关关闭的情况

```
客服进入会话
  → 检查开关
  → 全局或会话开关关闭
  → 不触发翻译，不显示翻译内容
```

## 六、完整流程图

### 用户进入会话的翻译流程

```
用户进入会话
    ↓
检查全局开关
    ↓
  关闭 → 不翻译，结束
    ↓ 开启
检查会话开关
    ↓
  关闭 → 不翻译，结束
    ↓ 开启
拉取消息列表
    ↓
筛选未翻译的消息（translation 字段为空）
    ↓
  无未翻译消息 → 直接显示，结束
    ↓ 有未翻译消息
收集消息 ID 列表
    ↓
发送批量翻译请求
    ↓
前端显示"翻译中..."
    ↓
后端分批处理（每批 50 条，最新优先）
    ↓
逐条翻译完成 → 推送结果
    ↓
前端接收推送 → 更新显示翻译结果
    ↓
全部完成
```

### 新消息到达的翻译流程（现有逻辑）

```
新消息到达
    ↓
检查全局开关 AND 会话开关
    ↓
  任一关闭 → 不翻译
    ↓ 都开启
  用户在会话中？
    ↓
  是 → 立即翻译 → 显示"翻译中..." → 显示结果
    ↓
  否 → 不翻译（等待用户进入会话时触发）
```

## 七、批量翻译参数

### 每批翻译条数：50 条

**为什么是 50 条？**
- 平衡翻译速度和系统压力
- 每批翻译耗时约 2-3 秒
- 用户可以快速看到最新消息的翻译结果

### 翻译优先级：最新消息优先

**处理顺序：**
1. 按消息时间倒序排列（最新的在前）
2. 每 50 条为一批
3. 第一批（最新 50 条）优先翻译
4. 第二批、第三批...依次处理

**示例：**
- 用户有 200 条未读消息
- 分为 4 批：第 1 批（最新 50 条）、第 2 批（次新 50 条）、第 3 批、第 4 批
- 第 1 批 2-3 秒完成，第 2 批 4-6 秒完成，依此类推
- 用户最先看到最新消息的翻译结果


## 九、大量未读消息的处理

### 问题

用户有几百上千条未读消息时，如何处理？

### 方案

**全部自动翻译，最新的优先**

1. 用户进入会话
2. 所有未翻译的消息都显示"翻译中..."
3. 最新的消息先完成翻译（1-2 秒）
4. 较早的消息陆续完成
5. 全部完成后，用户滚动查看时不会再看到"翻译中"

### 为什么不只翻译可见的消息？

- 钉钉/飞书的做法是全部翻译，不只是可见区域
- 用户滚动时体验更好，不会看到 loading
- 实现更简单

---

## 十、风险与应对

| 风险 | 应对 |
|-----|------|
| 大量消息翻译慢 | 最新消息优先翻译 |
| 翻译失败 | 显示"翻译失败，点击重试" |
| 成本增加 | 只翻译用户实际查看的会话 |
