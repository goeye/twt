# 消息自动翻译优化产品方案

## 一、当前问题

用户在会话中时，新消息会自动翻译。但用户离开会话后收到的消息，再次进入会话时不会自动翻译，需要手动点击翻译按钮，体验差。

## 二、优化目标

用户进入会话时，所有未翻译的消息自动开始翻译，无需手动操作。

---

## 三、钉钉/飞书的实际做法

### 核心逻辑

**触发时机：用户进入会话时**

1. 用户打开会话
2. 系统检查哪些消息还没有翻译
3. 自动开始翻译这些消息
4. 消息先显示"翻译中..."
5. 翻译完成后自动更新为翻译结果

### 关键特点

- 不是后台提前翻译好，而是用户进入会话时才触发
- 翻译结果会保存，下次进入直接显示，不会重复翻译
- 即使有很多未读消息，也会全部自动翻译

---

## 四、翻译开关逻辑

### 两级开关设计

**1. Copilot 全局开关（系统级）**
- 位置：系统设置 → Copilot 设置 → 消息翻译开关
- 作用：控制整个系统的翻译功能
- 优先级：最高

**2. 会话开关（会话级）**
- 位置：会话页面右上角工具栏
- 作用：控制当前会话是否翻译
- 优先级：次于全局开关

### 开关判断逻辑

```
是否翻译 = 全局开关 AND 会话开关

全局开关关闭 → 不翻译（无论会话开关状态）
全局开关开启 + 会话开关关闭 → 不翻译
全局开关开启 + 会话开关开启 → 翻译
```

### 开关状态表

| 全局开关 | 会话开关 | 是否翻译 |
|---------|---------|---------|
| 关闭 | 关闭 | ❌ 不翻译 |
| 关闭 | 开启 | ❌ 不翻译 |
| 开启 | 关闭 | ❌ 不翻译 |
| 开启 | 开启 | ✅ 翻译 |

## 五、产品方案

### 场景 1：客服在会话中（保持现有逻辑）

```
新消息到达 → 检查开关 → 自动翻译 → 显示翻译结果
```

### 场景 2：客服不在会话中（新增逻辑）

```
客服离开会话
  → 有新消息到达
  → 暂不翻译

客服再次进入会话
  → 检查全局开关和会话开关
  → 如果都开启，检查未翻译的消息
  → 自动开始翻译
  → 显示"翻译中..."
  → 翻译完成后显示结果
```

### 场景 3：客服再次进入已翻译过的会话

```
客服进入会话
  → 消息已有翻译结果
  → 直接显示翻译内容（无需等待）
```

### 场景 4：开关关闭的情况

```
客服进入会话
  → 检查开关
  → 全局或会话开关关闭
  → 不触发翻译，不显示翻译内容
```

### 场景 5：翻译过程中关闭开关

```
用户在翻译过程中关闭全局开关或会话开关
  → 立即停止后续翻译请求
  → 已发送的翻译请求继续完成
  → 已显示"翻译中..."的消息保持状态，不更新为翻译结果
  → 用户重新开启开关时，未完成的消息重新触发翻译
```

### 场景 6：翻译过程中离开会话

```
用户在翻译过程中离开会话
  → 后台翻译继续进行
  → 翻译结果正常保存到数据库
  → 用户再次进入会话时，已翻译的消息直接显示结果，未翻译的消息继续翻译
```

### 场景 7：会话列表中的翻译展示

```
会话列表中的最新消息预览
  → 如果该消息已有翻译结果，显示译文
  → 如果该消息未翻译，显示原文
  → 不在会话列表中显示"翻译中..."状态
```

### 场景 8：特殊消息内容处理

```
特殊消息内容处理规则：
  → 消息内容为空：不翻译，不显示翻译区域
  → 消息内容只有表情符号：不翻译，不显示翻译区域
  → 消息内容只有数字：不翻译，不显示翻译区域
  → 消息内容包含代码块：整体翻译，代码块内容保持原样
  → 消息内容为图片/文件：不翻译，不显示翻译区域
```

## 六、完整流程图

### 用户进入会话的翻译流程

```
用户进入会话
    ↓
检查全局开关
    ↓
  关闭 → 不翻译，结束
    ↓ 开启
检查会话开关
    ↓
  关闭 → 不翻译，结束
    ↓ 开启
拉取消息列表
    ↓
筛选未翻译的消息
    ↓
  无未翻译消息 → 直接显示，结束
    ↓ 有未翻译消息
收集消息 ID 列表
    ↓
发送批量翻译请求
    ↓
前端显示"翻译中..."
    ↓
后端分批处理（每批 50 条，最新优先）
    ↓
逐条翻译完成 → 通过 WebSocket 推送结果
    ↓
前端接收推送 → 更新显示翻译结果
    ↓
全部完成
```

### 新消息到达的翻译流程（现有逻辑）

```
新消息到达
    ↓
检查全局开关 AND 会话开关
    ↓
  任一关闭 → 不翻译
    ↓ 都开启
  用户在会话中？
    ↓
  是 → 立即翻译 → 显示"翻译中..." → 显示结果
    ↓
  否 → 不翻译（等待用户进入会话时触发）
```

## 七、批量翻译参数

### 每批翻译条数：50 条

**为什么是 50 条？**
- 平衡翻译速度和系统压力
- 每批翻译耗时 2-3 秒（实际耗时取决于网络状况和翻译服务响应速度）
- 用户可以在 2-3 秒内看到最新消息的翻译结果

### 翻译优先级：最新消息优先

**处理顺序：**
1. 按消息时间倒序排列（最新的在前）
2. 每 50 条为一批
3. 第一批（最新 50 条）优先翻译
4. 第二批、第三批...依次处理

**示例：**
- 用户有 200 条未读消息
- 分为 4 批：第 1 批（最新 50 条）、第 2 批（次新 50 条）、第 3 批、第 4 批
- 第 1 批 2-3 秒完成，第 2 批 4-6 秒完成，依此类推
- 用户最先看到最新消息的翻译结果

### 并发控制

**多会话并发处理：**
- 每个会话的翻译请求独立处理，互不影响
- 同一会话的重复翻译请求会取消前一次请求
- 用户离开会话后再次进入，会检查上次请求状态：
  - 若上次请求仍在进行，继续等待
  - 若上次请求已完成，直接显示结果
  - 若上次请求已取消或失败，重新发起请求

## 八、交互设计

### 翻译状态

**状态 1：翻译中**
```
消息原文下方显示"翻译中..."
```

**状态 2：翻译完成**
```
消息原文下方显示翻译结果
```

**状态 3：翻译失败**
```
消息原文下方显示"翻译失败，点击重试"
```

**重试交互：**
- 点击"翻译失败，点击重试"后：
  - 立即显示"翻译中..."
  - 重试成功后显示翻译结果，无 Toast 提示
  - 重试失败后继续显示"翻译失败，点击重试"
  - 无重试次数限制，用户可多次重试

### 状态转换规则

**翻译中 → 翻译完成：**
- 后端返回翻译结果（HTTP 200）
- 前端接收到 WebSocket 推送消息
- 翻译结果非空

**翻译中 → 翻译失败：**
- 后端返回错误（HTTP 4xx/5xx）
- 请求超时（30 秒）
- 网络异常
- 翻译结果为空

**翻译失败后：**
- 不自动重试，需用户手动点击重试

## 九、大量未读消息的处理

### 问题

用户有几百上千条未读消息时，如何处理？

### 方案

**全部自动翻译，最新的优先**

1. 用户进入会话
2. 所有未翻译的消息都显示"翻译中..."
3. 最新的消息先完成翻译（1-2 秒）
4. 较早的消息陆续完成
5. 全部完成后，用户滚动查看时不会再看到"翻译中"

### 为什么不只翻译可见的消息？

- 钉钉/飞书的做法是全部翻译，不只是可见区域
- 用户滚动时体验更好，不会看到 loading
- 实现更简单

---

## 十、风险与应对

| 风险 | 应对 |
|-----|------|
| 大量消息翻译慢 | 最新消息优先翻译 |
| 翻译失败 | 显示"翻译失败，点击重试" |
| 网络异常 | 请求超时 30 秒后显示"翻译失败，点击重试"，不自动重试 |
| 成本增加 | 只翻译用户实际查看的会话 |
