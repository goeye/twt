# 消息自动翻译优化 PRD

## 一、需求背景

客服在会话/聊天中时，新消息会自动翻译。但客服离开会话/聊天后收到的消息，再次进入会话/聊天时不会自动翻译，需要手动点击翻译按钮。

## 二、功能目标

客服进入会话/聊天时，视口内可见的未翻译消息自动翻译。客服滚动查看时，新进入视口的消息也自动翻译。

---

## 三、翻译开关

### 3.1 开关层级

**全局开关**
- 位置：系统设置 → Copilot 设置 → 聊天翻译开关
- 作用：控制整个系统的翻译功能

**会话开关**
- 位置：会话/聊天输入框工具栏
- 作用：控制当前会话/聊天是否翻译

### 3.2 开关判断规则

翻译生效条件：全局开关开启 且 会话开关开启

| 全局开关 | 会话开关 | 是否翻译 |
|---------|---------|---------|
| 关闭 | 关闭 | 否 |
| 关闭 | 开启 | 否 |
| 开启 | 关闭 | 否 |
| 开启 | 开启 | 是 |

---

## 四、功能场景

### 4.1 客服在会话中收到新消息

**触发条件**：客服正在查看会话，新消息到达

**执行逻辑**：
1. 检查全局开关和会话开关是否都开启
2. 如果都开启，新消息自动翻译
3. 如果任一关闭，不翻译

### 4.2 客服进入会话

**触发条件**：客服打开会话页面

**执行逻辑**：
1. 检查全局开关和会话开关是否都开启
2. 如果任一关闭，不翻译，结束
3. 如果都开启，检测视口内可见的消息
4. 筛选出未翻译的消息（最多 50 条）
5. 如果没有未翻译消息，直接显示，结束
6. 如果有未翻译消息，发送批量翻译请求
7. 前端显示"翻译中..."
8. 翻译完成后通过 WebSocket 推送结果
9. 前端更新显示翻译结果

### 4.3 客服滚动查看消息

**触发条件**：客服在会话中滚动消息列表

**执行逻辑**：
1. 监听滚动事件（防抖 300ms）
2. 检测新进入视口的消息
3. 筛选出未翻译的消息
4. 如果有未翻译消息，重复 4.2 的步骤 6-9

### 4.4 客服再次进入已翻译过的会话

**触发条件**：客服打开之前翻译过的会话

**执行逻辑**：
1. 加载消息列表
2. 已有翻译结果的消息直接显示译文
3. 无需等待或显示加载状态

### 4.5 翻译过程中关闭开关

**触发条件**：客服在翻译过程中关闭全局开关或会话开关

**执行逻辑**：
1. 立即停止后续翻译请求
2. 已发送的翻译请求继续完成
3. 客服重新开启开关时，新的消息才会继续翻译

### 4.6 翻译过程中离开会话

**触发条件**：客服在翻译过程中切换到其他会话或关闭会话

**执行逻辑**：
1. 停止当前翻译请求
2. 已发送的翻译请求继续完成
3. 翻译结果保存到数据库
4. 客服再次进入会话时，已翻译的消息直接显示结果
5. 视口内未翻译的消息重新触发翻译

### 4.7 会话列表中的消息预览

**显示规则**：
- 如果最新消息已有翻译结果，显示译文
- 如果最新消息未翻译，显示原文
- 不显示"翻译中..."状态

---

## 五、翻译状态

### 5.1 状态定义

**翻译中**
- 显示文案："翻译中..."
- 触发时机：发送翻译请求后，收到翻译结果前

**翻译完成**
- 显示内容：翻译结果文本
- 触发时机：收到翻译结果且结果非空

**翻译失败**
- 显示文案："翻译失败，点击重试"
- 触发时机：请求超时（30 秒）、网络异常、后端返回错误、翻译结果为空

### 5.2 状态转换规则

**翻译中 → 翻译完成**
- 后端返回成功（HTTP 200）
- 前端收到 WebSocket 推送消息
- 翻译结果非空

**翻译中 → 翻译失败**
- 后端返回错误（HTTP 4xx/5xx）
- 请求超时（30 秒）
- 网络异常
- 翻译结果为空

**翻译失败 → 翻译中**
- 客服点击"翻译失败，点击重试"
- 立即显示"翻译中..."
- 重新发送翻译请求

### 5.3 重试规则

- 翻译失败后不自动重试
- 客服手动点击重试
- 无重试次数限制
- 重试成功后显示翻译结果，无提示
- 重试失败后继续显示"翻译失败，点击重试"

## 六、技术参数

### 6.1 批量翻译

**每批翻译条数**：50 条

**触发机制**：
- 进入会话时检测视口内可见消息
- 滚动时检测新进入视口的消息
- 使用 Intersection Observer API 监听消息元素可见性

**处理逻辑**：
- 收集视口内所有未翻译的消息 ID（最多 50 条）
- 发送批量翻译请求
- 如果视口内超过 50 条未翻译消息，优先翻译最新的 50 条
- 客服继续滚动时，剩余消息在进入视口时触发新的翻译请求

### 6.2 并发控制

**多会话处理**：
- 每个会话的翻译请求独立处理，互不影响

**滚动防抖**：
- 滚动停止 300ms 后才触发翻译检测
- 短时间内多次滚动只发送一次翻译请求

### 6.3 超时设置

**请求超时时间**：30 秒

**超时处理**：
- 显示"翻译失败，点击重试"
- 不自动重试

---

## 七、边界情况

### 7.1 大量未读消息

**场景**：客服有几百上千条未读消息

**处理方式**：
- 只翻译视口内可见的消息（最多 50 条）
- 视口外的消息保持未翻译状态
- 客服滚动时，新进入视口的消息自动翻译

### 7.2 翻译失败

**失败原因**：
- 网络异常
- 请求超时（30 秒）
- 后端返回错误

**处理方式**：
- 显示"翻译失败，点击重试"
- 不自动重试
- 客服手动点击重试

### 7.3 开关状态变更

**场景**：翻译过程中关闭开关

**处理方式**：
1. 立即停止后续翻译请求
2. 已发送的翻译请求继续完成
3. 客服重新开启开关时，新的消息才会继续翻译

### 7.4 离开会话

**场景**：翻译过程中切换到其他会话

**处理方式**：
- 停止当前翻译请求
- 已发送的请求继续完成
- 翻译结果保存到数据库
- 再次进入会话时，已翻译的消息直接显示
- 视口内未翻译的消息重新触发翻译
