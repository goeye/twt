# 生成 PRD 文档

根据源码自动生成产品需求文档（PRD），保存至 `docs/` 目录。

调用方式：`/prd 功能名称`（如 `/prd 角色权限管理`、`/prd 访客标签`）

参数：$ARGUMENTS

## 步骤

### 第零阶段：加载记忆上下文（不可跳过）

0. **必须读取以下记忆文件**，它们记录了用户在历次会话中确认的需求结论和行为规范：
   - `memory/decisions-*.md`：与该功能相关的产品决策记录（**优先级高于源码**，源码可能尚未更新）
   - `memory/feedback_prd_writing.md`：PRD 编写规范（禁止项 + 必须项）
   - `memory/feedback_ui_global.md`：全局 UI 约定（Toast 文案、弹窗规范等）

   若决策记录与源码冲突，以决策记录为准，并在 PRD 中标注差异。

### 第一阶段：定位源码

1. 从参数中获取功能名称（即 `$ARGUMENTS`）。若参数为空，询问用户
2. 运行 `git log --oneline -30` 和 `git diff --stat HEAD~10` 定位近期与该功能相关的改动
3. 使用 Glob / Grep 搜索 `apps/web-agent/src/` 和 `packages/ui-agent/src/` 下相关文件

### 第二阶段：深度阅读

4. 逐一阅读**核心功能页面**，提取：
   - 数据结构（接口定义、mock 数据、枚举值）
   - 操作流程（CRUD 入口、路由跳转、弹窗触发）
   - 校验规则（表单验证、maxlength、必填项、禁用条件）
   - 文案字符串（弹窗标题、描述、Toast）
   - 保存方式（按钮提交 / 失焦保存 / 实时保存）

5. **全局影响面扫描**（不可跳过）：
   用 Grep 搜索核心实体名称在**整个项目**中的引用，找出所有关联页面。逐一检查：
   - 其他页面/模块中该实体的展示和操作
   - 不同用户角色下的差异行为
   - 列表页中的状态标签、操作菜单差异
   - 路由守卫、权限校验、403 页面等拦截点

### 第三阶段：排查清单

6. 在写 PRD 之前，对照以下清单逐项排查，确认无遗漏：

   **A. 实体生命周期**：所有状态 → 每个状态下的可用操作 → 状态流转路径 → 不可逆操作

   **B. 资源约束**：配额/名额 → 各状态是否占用配额 → 频率限制/冷却时间 → 数量上限

   **C. 关联页面**：其他页面的 UI 变化 → 列表标签/菜单更新 → 详情页字段变更 → 角色差异

   **D. 权限正反面**：具备权限时的行为 → 不具备权限时的表现（隐藏/置灰/拦截/403）

   **E. 外部触点**：官网/营销页面更新 → 定价/套餐对比 → 帮助文档

   **F. 输入校验闭环**（源码未定义时必须询问用户）：
   - 必填/选填、长度上限、格式要求、唯一性、允许字符范围
   - 长文本处理（换行/截断/省略号）、空值显示规则
   - CRUD 反馈闭环：成功 Toast（遵循全局规范）、失败提示、取消行为、二次确认弹窗

   **G. 列表页规范**：
   - 默认排序字段和方向 → 可排序列 → 分页规则 → 搜索/筛选默认值
   - 日期范围：默认范围、最大跨度、边界日包含规则

   **H. 口径一致性**：
   - 筛选项与表格值域是否对应 → 枚举命名是否统一 → 多规则同时命中的展示策略 → 幂等性

### 第四阶段：撰写文档

7. 按下方「PRD 文档大纲」撰写完整文档
8. 保存至 `docs/PRD-{功能名称}.md`
9. 告知用户文档路径

## 写作规范

1. 全文使用**中文**，**严禁英文代码术语**（变量名、状态名、注释风格）
2. **严禁 UI 样式描述**：不得出现颜色、字体、尺寸、圆角、阴影、边框、间距等视觉设计。详见 `memory/feedback_prd_writing.md`
3. **只写最终结果**：已排除的功能不要提及（包括「排除项」说明），就像从来不存在
4. **避免冗余**：同一规则只在最合适的章节写一次，其他处用「详见 x.x 节」引用
5. 所有数据从源码提取，不凭空编造
6. 章节无内容可省略，不填充占位文字
7. 功能未完整实现时，在概述标注「状态：部分实现」并说明缺失部分
8. **写完后必须回顾第三阶段排查清单（A-H），逐项确认无遗漏后再输出**

## 需求决策记录规则

在日常会话中（不仅限于 /prd），遵循以下规则：

1. **自动记录**：会话涉及需求讨论或代码改动时，将关键结论写入 `memory/decisions-{模块名}.md`
2. **主动记录**：用户说「记一下」「记录」时立刻写入
3. **冲突标注**：新结论覆盖旧决策时，标注「覆盖：原结论是 xxx」
4. **格式**：`- {日期}：{结论}`，按功能分组
