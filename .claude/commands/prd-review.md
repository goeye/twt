# PRD 文档校验

深度校验 PRD 文档的完整性、一致性和清晰度，发现歧义、矛盾、遗漏和边界问题。

调用方式：`/prd-review docs/PRD-功能名称.md`

参数：$ARGUMENTS（PRD 文档路径）

## 校验步骤

### 第一步：读取文档

1. 从参数中获取 PRD 文档路径。若参数为空，列出 `docs/` 目录下所有 PRD 文档供用户选择
2. 完整读取 PRD 文档内容

### 第二步：执行 8 大维度校验

逐项检查以下维度，发现问题立即记录，**不要跳过任何检查项**：

#### A. 输入组件完整性校验

针对文档中提到的每个用户输入字段（文本框、下拉框、单选、多选、日期选择器等），检查是否明确定义：

- [ ] **必填/选填**状态
- [ ] **字符长度上限**（如"最多 50 字符"）
- [ ] **格式校验规则**（如邮箱格式、数字范围、正则表达式）
- [ ] **唯一性约束**（是否允许重复，范围是全局还是局部）
- [ ] **允许字符范围**（是否限制特殊字符、空格处理规则）
- [ ] **长文本处理方式**（超出显示区域时：换行/截断/省略号）
- [ ] **空值显示规则**（字段为空时显示什么）
- [ ] **默认值**（如有）

**问题示例**：
- ❌ "用户填写备注名称" → 缺少长度限制、是否必填
- ✅ "备注名称选填，最多 50 字符，超出部分截断并显示省略号，为空时显示 '--'"

#### B. 交互行为明确性校验

检查每个用户操作是否定义清晰：

- [ ] **按钮禁用/启用条件**（什么情况下按钮不可点击）
- [ ] **加载状态**（提交/保存时是否有 loading 状态）
- [ ] **成功反馈**（Toast 文案是否符合全局规范：新增成功/保存成功/删除成功/编辑成功）
- [ ] **失败反馈**（校验失败、网络错误、业务错误的提示文案）
- [ ] **二次确认弹窗**（删除、清空等不可逆操作的弹窗标题、描述、按钮文案）
- [ ] **取消/放弃行为**（是否有未保存提示、提示文案）
- [ ] **页面跳转时机**（操作成功后是否跳转、跳转到哪里）

**问题示例**：
- ❌ "点击删除按钮删除角色" → 缺少二次确认弹窗定义
- ✅ "点击删除按钮弹出确认弹窗，标题'确认删除'，描述'删除后不可恢复'，按钮'取消'和'确认删除'"

#### C. 业务逻辑一致性校验

检查文档中的业务规则是否前后一致：

- [ ] **状态定义一致性**（同一状态在不同章节的描述是否一致）
- [ ] **枚举值一致性**（筛选项、表格列、详情页的枚举值是否完全对应）
- [ ] **数量/配额规则一致性**（不同位置提到的上限是否一致）
- [ ] **权限规则一致性**（权限章节与功能描述中的权限要求是否一致）
- [ ] **数据流转逻辑**（创建 → 编辑 → 删除的数据变化是否合理）

**问题示例**：
- ❌ 第 3.3.1 节说"最多创建 10 个角色"，第 6 节约束表格说"最多 5 个自定义角色"
- ❌ 筛选项有"待激活/已激活/已禁用"，但表格状态列只显示"正常/禁用"

#### D. 状态机完整性校验

如果功能涉及状态流转，检查：

- [ ] **所有状态是否都有定义**（含义、触发条件）
- [ ] **状态转换路径是否完整**（每个状态可以转换到哪些状态）
- [ ] **状态转换条件是否明确**（什么操作触发状态变化）
- [ ] **终态是否明确**（哪些状态是不可逆的）
- [ ] **初始状态是否明确**（实体创建时的默认状态）
- [ ] **状态与操作的对应关系**（每个状态下允许/禁止哪些操作）

**问题示例**：
- ❌ 状态机图显示"待激活 → 已激活 → 已禁用"，但未说明"已禁用"能否恢复到"已激活"
- ❌ 提到"已删除"状态，但状态机图中没有这个状态

#### E. 边界情况覆盖度校验

检查是否考虑了常见边界场景：

- [ ] **空数据状态**（列表为空、搜索无结果时显示什么）
- [ ] **数量上限**（达到上限后的提示和限制行为）
- [ ] **数量下限**（如"至少保留 1 个"的场景）
- [ ] **重复提交**（相同操作重复执行的幂等性规则）
- [ ] **并发冲突**（多人同时编辑同一数据的处理方式）
- [ ] **依赖缺失**（关联数据被删除后的处理）
- [ ] **权限变更**（用户权限被收回后的降级表现）
- [ ] **网络异常**（请求失败、超时的处理）

**问题示例**：
- ❌ 只描述了"删除角色"，未说明"角色下有客服时能否删除"
- ❌ 只描述了"搜索客服"，未说明"搜索无结果时显示什么"

#### F. 跨模块影响面校验

检查功能改动是否考虑了对其他模块的影响：

- [ ] **列表页改动**（该实体在其他列表中的展示是否需要更新）
- [ ] **详情页改动**（其他模块查看该实体详情时是否需要更新）
- [ ] **关联数据改动**（删除/修改该实体时，关联数据如何处理）
- [ ] **权限联动**（权限变更是否影响其他模块的可见性）
- [ ] **配额联动**（该实体的增删是否影响配额计算）

**问题示例**：
- ❌ PRD 只描述了"角色管理"页面，未提及"客服列表"中角色列的显示规则
- ❌ 未说明"删除角色后，该角色下的客服如何处理"

#### G. 歧义与模糊表述检查

检查文档中是否存在可能产生歧义的表述：

- [ ] **"可以"/"应该"等模糊词**（改为明确的"必须"/"禁止"/"允许"）
- [ ] **"合适的"/"适当的"等主观词**（改为具体的数值或规则）
- [ ] **"等"/"相关"等省略词**（补充完整列表）
- [ ] **"一般"/"通常"等不确定词**（明确具体场景）
- [ ] **多重否定**（改为肯定表述）
- [ ] **代词指代不清**（"该功能"/"此时"等，明确指代对象）

**问题示例**：
- ❌ "用户可以填写相关信息" → 哪些信息？必填还是选填？
- ❌ "系统会进行适当的校验" → 具体校验什么？规则是什么？
- ❌ "一般情况下显示角色名称" → 什么是一般情况？特殊情况呢？

#### H. UI 样式描述检查（禁止项）

检查文档中是否违规出现了 UI 样式描述：

- [ ] **颜色描述**（如"蓝色按钮"、"灰色文字"、"#333333"）
- [ ] **字体描述**（如"16px 字体"、"加粗"、"font-weight: 500"）
- [ ] **尺寸描述**（如"高度 32px"、"宽度 200px"、"圆角 4px"）
- [ ] **布局描述**（如"左右布局"、"flex 布局"、"间距 16px"）
- [ ] **样式描述**（如"阴影效果"、"边框样式"、"渐变背景"）

**问题示例**：
- ❌ "保存按钮为蓝色背景，白色文字，高度 32px"
- ❌ "标题使用 16px 加粗字体，颜色 #222222"

## 注意事项

- 校验时必须**逐字逐句**阅读 PRD 文档，不要遗漏任何章节
- 发现问题时必须**指出具体位置**（章节号、段落）和**具体内容**
- 每个问题必须给出**明确的改进建议**，不要只说"需要补充"
- 问题按严重程度分级：🔴 严重（阻塞）、🟡 中等（建议）、🟢 轻微（优化）
- 严重问题：缺少关键规则、逻辑矛盾、边界遗漏
- 中等问题：表述不清、一致性问题、部分边界未覆盖
- 轻微问题：措辞优化、格式问题、非关键 UI 描述
- 校验报告必须**客观中立**，基于事实而非主观判断
- 如果 PRD 文档质量很高，也要如实说明，不要为了找问题而找问题
