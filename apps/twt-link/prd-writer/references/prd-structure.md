# PRD Structure

正式输出 PRD 时，必须按以下结构执行。

## 文档级结构

先输出文档级固定章节，再按实际层级输出业务主体章节。
若无模块层，则从页面层开始；若无页面层，则直接从功能层开始。

### 1 背景与目标（文档级固定章节，必须完整输出）

#### 1.1 业务背景
- 当前业务现状与痛点。
- 触发本次需求的原因（用户反馈、经营目标、流程效率、合规要求等）。
- 影响范围（角色、页面、核心流程）。

#### 1.2 目标
- 本期目标（至少 1 条，可量化）。
- 每条目标需包含：目标描述、衡量口径、目标值或期望区间（不明确则`（待确认）`）。

#### 1.3 验收指标
- 每个目标至少对应 1 条验收指标。
- 验收指标需包含：指标名称、计算口径、统计周期、验收阈值、数据来源（不明确则`（待确认）`）。
- 无法量化时，给出可执行的定性验收标准并标注`（待确认）`。

## 页面级章节

存在页面层时，对每个页面仅输出一次“页面整体概览”：
- 描述页面结构区块与信息布局。
- 输出页面结构 Mermaid 图（业务语义命名）。

## 功能级章节

对每一个功能（即每个 `f`），都必须完整输出以下章节：

### `f.1 功能定义`
- 用字段标签式表达核心信息。
- 至少输出：`**功能描述：** ...`、`**用户场景：** ...`、`**功能入口与触发方式：** ...`

### `f.2 交互流程`
- 用步骤描述用户动作、系统处理、反馈结果。
- 明确角色动作、状态路径、分支条件。
- 输出流程 Mermaid 图（业务语义命名）。

### `f.3 前置条件`（必填）
- 以字段标签式表达前置条件。
- 检查维度包括：登录状态、角色与权限、前置业务条件、依赖配置或前序步骤。
- 写法示例：`**登录状态：** ...`、`**角色与权限：** ...`

### `f.4 输入规则`（必填）
- 若代码或需求中有明确证据，按证据输出；若用户期望完整评审稿而证据缺失，可基于常见产品设计补全具体规则值，并标注`（待确认）`。
- 覆盖维度：
  - 各输入项必填/选填
  - 各输入项输入阈值（如长度、数值范围等）
  - 各输入项输入格式
  - 各输入项默认值/取值范围
  - 输入项有状态或枚举值时，列出所有状态或枚举值及其含义
- 每条规则应保留字段名，使用`**字段名：** 规则说明`格式。
- 优先输出已确认信息，例如控件类型、是否必填、默认值、可选值；阈值和格式缺失时，补充具体规则值并标注`（待确认）`。
- 正确示范：`**物品名称：** 文本输入，支持模糊搜索，选填。`
- 正确示范：`**业务状态：** 下拉单选，选填，可选值为在用/停用/已作废。`
- 正确示范：`**电话/宽带号码：** 文本输入，必填，最多支持50个字符（待确认）。`
- 错误示范：`**电话/宽带号码：** 输入阈值与格式规则（待确认）。`

### `f.5 校验规则`（必填）
- 只描述业务规则，不涉及技术实现方式。
- 如果没有对应规则，可省略无内容项，但不得输出占位词。
- 覆盖维度：
  - 提交前校验条件
  - 业务约束校验条件
  - 校验失败提示文案
  - 校验失败后的可执行动作

### `f.6 业务规则`（必填）
- 如果没有对应规则，可省略无内容项，但不得输出占位词。
- 覆盖维度：
  - 数据生成/修改/删除/变更规则
  - 状态定义与状态转换条件
  - 排序/筛选/分页/搜索规则
  - 权限差异下的行为规则

### `f.7 展示与交互状态规则`（必填）
- 如果没有对应规则，可省略无内容项，但不得输出占位词。
- 覆盖维度：
  - 字段展示与映射规则
  - 格式化规则（时间、金额、脱敏等）
  - 空态、加载态、禁用态、只读态、隐藏态触发条件
  - 成功/失败后的反馈方式（提示、跳转、刷新、局部更新）

### `f.8 异常处理`（必填）
- 如果没有对应规则，可省略无内容项，但不得输出占位词。
- 覆盖维度：
  - 无数据
  - 请求失败
  - 无权限
  - 重复提交
  - 超时/中断
- 每类异常都要写“提示文案 + 用户可执行后续动作”。

### `f.9 后置条件`
- 功能操作完成后，系统和数据应达到的状态，包括但不限于：
  - 数据新增、修改或删除的结果与归属
  - 状态变化（如已提交、已审批、已驳回、已归档等）
  - 通知是否发送、发送给谁、以何种方式
  - 页面是否跳转、是否刷新列表、是否记录操作日志
  - 对其他模块、报表或统计数据的影响

### `f.10 补充条件`
- 如果没有对应规则，可省略无内容项，但不得输出占位词。
- 覆盖维度：
  - 限制条件与例外情况
  - 特殊角色豁免策略
  - 历史数据兼容策略（如不明确则`（待确认）`）

## 信息架构输出模板

### `# 1 信息架构草案（待确认）`

业务主体按实际层级选择其一：
- `## 2 模块名称` -> `### 2.1 页面名称` -> `#### 2.1.1 功能名称`
- `## 2 页面名称` -> `### 2.1 功能名称`
- `## 2 功能名称`

功能节点下补充以下信息：
- 功能类型：
- 目标用户/角色：
- 入口与触发方式：
- 输出结果：

### `## 3 待你确认`
- 请回复：确认 / 补充
